# 방안 3: 예약 가능 시간대 사전 공지

> 상위 문서: [[문제 분석 및 해결 방안]]
> 난이도: 낮음 | 소요: ~1주 | 기대 효과: 중-높 | 우선순위: Tier 1 - 3순위

---

## 1. 핵심 가설

> "학생이 예약하려 들어왔다가 풀부킹을 발견하는" 부정적 경험을 제거하면, 같은 공급량에서도 체감 만족도가 올라가고 이탈이 줄어든다. 동시에 이른 시간 예약을 유도하여 예약 분산 효과도 기대할 수 있다.

**핵심 전환:**
```
AS-IS: 학생이 앱 진입 → 시간 선택 → "풀부킹" 발견 → 좌절/이탈
TO-BE: 학생이 알림 수신 → "내일 20시에 자리 있음" 확인 → 즉시 예약
```

---

## 2. 알림 설계

### 2-1. 발송 타이밍 옵션

| 옵션 | 시점 | 장점 | 단점 |
|------|------|------|------|
| A. 전날 저녁 21시 | D-1 21:00 | 다음날 계획 세울 때 확인 | 이후 변동 반영 안 됨 |
| B. 당일 아침 8시 | D-day 08:00 | 최신 정보, 당일 행동 유도 | 조기 마감 시 실망 |
| C. 전날 저녁 + 당일 아침 | 두 번 | 양쪽 장점 | 알림 피로 |

**권장: 옵션 A (전날 저녁 21시)** - MVP에서 1회로 시작, 데이터 보고 확장

### 2-2. 알림 콘텐츠 설계

**푸시 알림 (짧은 버전):**
```
📅 내일 예약 가능한 수업이 있어요!
19시, 20시, 21시에 자리가 있습니다.
[지금 예약하기]
```

**인앱 상세 버전 (딥링크 랜딩):**
```
┌─────────────────────────────────────────┐
│  📅 3/25(화) 예약 가능 시간              │
│                                          │
│  ✅ 19:00  빈자리 3개                    │
│  ✅ 20:00  빈자리 1개  ⚡ 곧 마감        │
│  ✅ 21:00  빈자리 2개                    │
│  ❌ 22:00  마감                          │
│                                          │
│  💡 일찍 예약할수록 원하는 튜터를          │
│     만날 확률이 높아요!                   │
│                                          │
│  [19시 예약하기] [20시 예약하기] [21시 예약하기]
└─────────────────────────────────────────┘
```

### 2-3. 개인화 레벨

| 레벨 | 내용 | 구현 복잡도 |
|------|------|-----------|
| L1 (MVP) | 전체 가용 시간대 안내 | 낮음 |
| L2 | 학생의 과거 예약 패턴 기반 선호 시간대만 안내 | 중 |
| L3 | 학생이 좋아하는 튜터의 빈자리만 안내 | 높음 |

**MVP는 L1으로 시작, 전환률 데이터 보고 L2 → L3 순서로 고도화**

---

## 3. "빈자리 N개" 계산 로직

### 3-1. 기본 로직

```
특정 시간대의 빈자리 수
  = 해당 시간대 오픈된 튜터 슬롯 수 - 예약된 수업 수
```

### 3-2. 고려사항

- **스냅샷 시점**: 알림 발송 시점의 가용 슬롯 기준
- **버퍼**: 알림 발송 후 ~ 학생 확인 사이에 다른 학생이 예약할 수 있음
  - "빈자리 1개"로 안내했는데 들어가보니 마감 → 오히려 역효과
  - **완화:** "빈자리 1개 → 곧 마감" 표시로 기대치 관리, 또는 1개일 때는 안내하지 않음
- **최소 빈자리 기준**: 빈자리 2개 이상인 시간대만 안내 (1개는 변동성 높음)

### 3-3. 바이링구얼 vs 전체 튜터

- 현재 JP 초급 학생은 바이링구얼 튜터만 예약 가능
- 빈자리 계산도 **바이링구얼 튜터 기준**으로 해야 정확
- 추후 Level Up 출시 후 비바이링구얼 포함으로 확장

## 4. 발송 대상 선정

### 4-1. 누구에게 보낼 것인가

모든 JP 학생에게 보내면 알림 피로 발생. 타겟팅 필요.

| 대상 | 기준 | 우선순위 |
|------|------|---------|
| 최근 7일 예약 실패 경험자 | 예약 시도 → 풀부킹 경험 | 최우선 |
| 활성 수강 중 + 최근 3일 수업 없음 | 레슨권 보유 + 비활성 | 높음 |
| 신규 구매자 (7일 이내) | 첫 수업 예약 전 | 높음 |
| 나머지 활성 수강생 | 정기적 수업 중 | 보통 |

**MVP: 상위 2개 그룹만 타겟팅 (예약 실패자 + 비활성 수강생)**

### 4-2. 발송 안 하는 경우

- 학생이 이미 내일 수업 예약이 있는 경우 → 불필요
- 학생이 알림 수신 거부한 경우
- 내일 예약 가능 시간대가 0개인 경우 → "예약 가능한 시간이 없습니다" 알림은 역효과

## 5. 부수 효과: 예약 분산

### 5-1. 현재 예약 패턴 (추정)

```
학생 예약 시점 분포:
  당일 오전: 10%
  당일 오후: 25%
  당일 저녁 (수업 직전): 40%  ← 대부분 여기서 풀부킹 경험
  전날 이전: 25%
```

### 5-2. 사전 공지 후 기대 패턴

```
  전날 저녁 (알림 수신 직후): 30%  ← 새로운 예약 피크
  당일 오전: 20%
  당일 오후: 20%
  당일 저녁: 20%
  전날 이전: 10%
```

- 전날 저녁에 예약이 분산되면, 당일 저녁의 풀부킹 체감이 줄어듦
- 실제 공급은 동일하지만 **예약 실패 빈도가 줄어드는 효과**

### 5-3. "이른 예약 = 좋은 선택" 프레이밍

- "일찍 예약할수록 원하는 튜터를 만날 확률이 높아요!"
- "지금 예약하면 인기 튜터 [이름] 선생님과 수업 가능!"
- 심리적 긴급성 + 보상 프레이밍 병행

## 6. 구현 범위

### Phase 1 (MVP, ~1주)
- [ ] 매일 21시 가용 시간대 스냅샷 생성 크론잡
- [ ] 타겟 학생 리스트 생성 로직 (예약 실패자 + 비활성)
- [ ] 푸시 알림 발송 (기존 인프라 활용)
- [ ] 딥링크: 알림 탭 → 예약 화면 (해당 날짜 프리셋)

### Phase 2 (~2주)
- [ ] 인앱 랜딩 페이지 (시간대별 빈자리 시각화)
- [ ] 개인화 L2: 학생 선호 시간대 기반 필터링
- [ ] 알림 오픈률/예약 전환률 트래킹

### 기술 스택
- 크론잡: 매일 21:00 실행
- 푸시: Firebase Cloud Messaging (기존 인프라)
- 데이터: 기존 예약 DB에서 시간대별 가용 슬롯 조회

## 7. 성공/실패 판단 기준

### 출시 후 2주 체크

| 지표 | 목표 | 실패 기준 |
|------|------|----------|
| 알림 오픈률 | 30%+ | 10% 미만 |
| 알림 → 예약 전환률 | 15%+ | 5% 미만 |
| 전날 예약 비율 변화 | +10pp | 변화 없음 |
| 풀부킹 체감 빈도 (웨이팅리스트 등록 수 기준) | -15% | 변화 없음 |

## 8. 리스크 및 완화

| 리스크 | 확률 | 영향 | 완화 방안 |
|--------|------|------|----------|
| 알림 피로 → 수신 거부 증가 | 중 | 높 | 주 3회 이하, 타겟팅 정밀화 |
| 알림 보고 갔는데 이미 마감 | 높 | 중 | 빈자리 2개+ 시간대만 안내, "곧 마감" 표시 |
| 특정 시간대에 예약 쏠림 (새로운 풀부킹) | 중 | 중 | 빈자리 수 표시로 분산 유도 |
| 비피크 시간대가 안내에서 배제 → 악순환 | 낮 | 낮 | 비피크에 여유 있을 때 "한적한 시간" 프레이밍 |
