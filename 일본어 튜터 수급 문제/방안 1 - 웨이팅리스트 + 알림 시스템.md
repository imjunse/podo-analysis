# 방안 1: 웨이팅리스트 + 알림 시스템

> 상위 문서: [[문제 분석 및 해결 방안]]
> 난이도: 낮음 | 소요: ~1-2주 | 기대 효과: 높음 | 우선순위: Tier 1 - 1순위

---

## 1. 핵심 가설

> "예약 불가 → 즉시 이탈"의 경로를 "예약 불가 → 대기 등록 → 알림 → 예약 성공"으로 전환하면, 풀부킹 상황에서도 학생 이탈률을 유의미하게 낮출 수 있다.

**측정 지표:**
- 웨이팅리스트 등록률 (풀부킹 경험 학생 중 등록한 비율)
- 알림 → 예약 전환률
- 웨이팅리스트 등록 학생의 7일/30일 리텐션 vs 미등록 학생
- booking 관련 환불 건수/금액 변화 (기준: 월 ~498만원 페이스)

---

## 2. 상세 UX 플로우

### 2-1. 풀부킹 감지 시점

학생이 예약 화면에 진입했을 때, 선택한 시간대의 모든 튜터가 예약 완료 상태인 경우 발동.

```
학생이 시간대 선택
    ↓
해당 시간대 가용 튜터 0명?
    ├─ Yes → 풀부킹 UI 표시
    └─ No  → 정상 예약 플로우
```

### 2-2. 풀부킹 UI (학생 화면)

```
┌─────────────────────────────────────────┐
│  😔 이 시간대는 현재 예약이 가득 찼어요   │
│                                          │
│  ┌──────────────────────────────────┐    │
│  │ 🔔 빈자리 알림 받기              │    │  ← Primary CTA
│  │    자리가 나면 바로 알려드릴게요     │    │
│  └──────────────────────────────────┘    │
│                                          │
│  ┌──────────────────────────────────┐    │
│  │ 🕐 다른 시간대 보기              │    │  ← Secondary
│  └──────────────────────────────────┘    │
│                                          │
│  ┌──────────────────────────────────┐    │
│  │ 📅 예약 가능한 시간 알림 받기     │    │  ← Tertiary (방안 5 통합)
│  │    매일 예약 가능 시간을 알려드려요  │    │
│  └──────────────────────────────────┘    │
└─────────────────────────────────────────┘
```

### 2-3. 알림 → 예약 전환 플로우

```
튜터 취소 or 새 스케줄 오픈 발생
    ↓
해당 시간대 웨이팅리스트 조회
    ↓
등록 순서대로 알림 발송 (1명씩)
    ↓
알림 받은 학생에게 5분간 예약 우선권 부여
    ├─ 5분 내 예약 → 완료, 다음 대기자에게 알림 불필요
    └─ 5분 초과 → 우선권 만료, 다음 대기자에게 알림 발송
```

### 2-4. 예약 우선권 5분의 근거

- 너무 짧으면 (1-2분): 알림 확인 전에 만료. 학생 불만 증가
- 너무 길면 (15-30분): 빈자리가 장기간 비어있게 되어 다른 학생 기회 차단
- **5분은 푸시 알림 확인 + 앱 진입 + 예약 완료에 필요한 최소 시간**
- A/B 테스트로 3분 vs 5분 vs 10분 비교 권장

## 3. 데이터 모델

### 3-1. 웨이팅리스트 테이블

```sql
CREATE TABLE lesson_waitlist (
    id              BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id         BIGINT NOT NULL,          -- 학생 ID
    target_date     DATE NOT NULL,            -- 원하는 수업 날짜
    target_hour     TINYINT NOT NULL,         -- 원하는 시간대 (0-23)
    language        VARCHAR(10) NOT NULL,     -- 'JP', 'EN'
    tutor_type      VARCHAR(20) DEFAULT 'bilingual', -- 'bilingual', 'any'
    status          VARCHAR(20) DEFAULT 'waiting',   -- 'waiting', 'notified', 'booked', 'expired', 'cancelled'
    notified_at     DATETIME NULL,            -- 알림 발송 시각
    priority_expires_at DATETIME NULL,        -- 예약 우선권 만료 시각
    created_at      DATETIME DEFAULT NOW(),
    updated_at      DATETIME DEFAULT NOW() ON UPDATE NOW(),

    INDEX idx_target (target_date, target_hour, language, status),
    INDEX idx_user (user_id, status)
);
```

### 3-2. 알림 로그 테이블

```sql
CREATE TABLE waitlist_notification_log (
    id              BIGINT PRIMARY KEY AUTO_INCREMENT,
    waitlist_id     BIGINT NOT NULL,
    notification_type VARCHAR(20) NOT NULL,   -- 'push', 'in_app'
    sent_at         DATETIME DEFAULT NOW(),
    opened_at       DATETIME NULL,
    result          VARCHAR(20) NULL,         -- 'booked', 'expired', 'failed'

    INDEX idx_waitlist (waitlist_id)
);
```

## 4. 핵심 로직

### 4-1. 트리거: 빈자리 발생 감지

빈자리가 발생하는 3가지 경우:

| 트리거 | 발생 빈도 | 처리 방식 |
|--------|----------|----------|
| 튜터가 수업 취소 | 높음 | 취소 이벤트 → 웨이팅리스트 체크 |
| 학생이 수업 취소 | 중간 | 취소 이벤트 → 웨이팅리스트 체크 |
| 튜터가 새 스케줄 오픈 | 낮음 | 스케줄 변경 이벤트 → 웨이팅리스트 체크 |

**처리 플로우 (의사코드):**

```
function onSlotAvailable(date, hour, language):
    waiters = getWaiters(date, hour, language, status='waiting')
    if waiters.isEmpty(): return

    firstWaiter = waiters.sortBy('created_at').first()

    sendPushNotification(firstWaiter, {
        title: "빈자리가 생겼어요!",
        body: "{date} {hour}시 수업 예약이 가능합니다. 5분 안에 예약하세요!",
        deepLink: "/booking?date={date}&hour={hour}&priority=true"
    })

    updateWaiter(firstWaiter, {
        status: 'notified',
        notified_at: now(),
        priority_expires_at: now() + 5.minutes
    })

    // 5분 후 만료 체크 스케줄
    scheduleJob(after: 5.minutes, action: checkPriorityExpired(firstWaiter.id))
```

### 4-2. 우선권 만료 처리

```
function checkPriorityExpired(waitlistId):
    waiter = getWaiter(waitlistId)
    if waiter.status == 'notified' AND now() > waiter.priority_expires_at:
        updateWaiter(waiter, { status: 'expired' })

        // 슬롯이 아직 비어있다면 다음 대기자에게
        if isSlotStillAvailable(waiter.target_date, waiter.target_hour):
            onSlotAvailable(waiter.target_date, waiter.target_hour, waiter.language)
```

### 4-3. 동시성 문제 처리

**문제:** 알림 받은 학생 A가 5분 내에 예약하려는데, 그 사이 일반 학생 B가 먼저 예약하면?

**해결:**
- 우선권 시간 동안 해당 슬롯에 "soft lock" 적용
- 다른 학생이 예약 시도 시: "잠시 후 다시 시도해주세요" 또는 "이 시간대에 대기 중인 학생이 있습니다"
- 구현: Redis 기반 TTL 5분짜리 lock key (e.g., `waitlist:lock:{date}:{hour}:{tutor_id}`)

**주의:** soft lock은 해당 튜터의 해당 시간 슬롯에만 적용. 다른 튜터가 같은 시간에 오픈하면 일반 예약 가능.

## 5. 엣지 케이스

### 5-1. 같은 시간대에 대기자가 50명이면?

- 실질적으로 1-3번째 대기자까지만 의미 있음
- **대기 순서 표시**: "현재 이 시간대에 3명이 대기 중입니다" → 기대치 관리
- **최대 대기 인원 제한**: 10명 이상이면 "이 시간대는 대기가 많습니다. 다른 시간대를 추천합니다" 안내

### 5-2. 학생이 여러 시간대에 동시 대기 등록하면?

- 허용하되, **최대 3개 시간대**로 제한
- 하나의 시간대에서 예약 성공 시 → 나머지 대기 자동 취소 (같은 날짜 기준)

### 5-3. 알림 받았는데 앱을 안 켜면?

- 5분 후 자동 만료 → 다음 대기자에게 넘어감
- 만료된 학생에게는 "아까 빈자리가 있었지만 다른 학생이 예약했어요. 다시 알림 받으시겠어요?" 후속 알림
- 재등록 원클릭 제공

### 5-4. 피크타임(19-22시)에 빈자리가 거의 안 나면?

- 웨이팅리스트만으로는 근본 해결 불가 (공급 자체가 부족)
- 이 경우 방안 2(주간 보너스)와의 시너지가 중요
- 웨이팅리스트 데이터가 "어떤 시간대가 가장 부족한지" 정량적으로 보여줌 → 방안 2의 타겟팅에 활용

### 5-5. 당일 수업 vs 다음날 수업

- **당일 수업**: 알림 → 5분 우선권이 적절
- **다음날 이후 수업**: 알림 → 30분~1시간 우선권으로 확대 검토 (긴급성이 낮으므로)
- MVP에서는 일단 5분으로 통일, 이후 데이터 보고 분리

## 6. 성공/실패 판단 기준

### 출시 후 2주 시점 체크

| 지표 | 목표 | 실패 기준 |
|------|------|----------|
| 웨이팅리스트 등록률 | 풀부킹 경험자의 30%+ | 10% 미만 |
| 알림 → 예약 전환률 | 40%+ | 15% 미만 |
| 5분 내 예약 성공률 | 60%+ | 30% 미만 |
| booking 관련 환불 추세 | 감소 시작 | 증가 지속 |

### 성공 시 확장 방향
- 튜터별 인기도 기반 "이 튜터에 빈자리 나면 알림" 기능
- 정기 수업 학생 우선 매칭

### 실패 시 피벗
- 5분 우선권 시간 조정 (3분/10분)
- 알림 채널 변경 (푸시 → 카카오톡/LINE)
- 근본적으로 공급이 부족한 것이므로 방안 2에 더 투자

## 7. 개발 범위 (MVP)

### Phase 1 (1주): 코어 기능
- [ ] 웨이팅리스트 DB 테이블 생성
- [ ] 풀부킹 시 "빈자리 알림 받기" UI 추가
- [ ] 빈자리 발생 시 알림 발송 로직 (크론잡 or 이벤트 기반)
- [ ] 예약 실패 이벤트 로깅 추가 (측정 인프라)

### Phase 2 (2주): 우선권 + 고도화
- [ ] 5분 예약 우선권 로직 (Redis soft lock)
- [ ] 대기 순서 표시 UI
- [ ] 만료 시 다음 대기자 자동 알림
- [ ] 대시보드: 시간대별 대기 인원, 전환률

### 기술 스택 참고
- 알림: 기존 푸시 알림 인프라 재사용
- 큐: 빈자리 이벤트 → 알림 발송 비동기 처리
- Lock: Redis SET NX EX 5분
- 모니터링: 기존 이벤트 로깅 파이프라인에 이벤트 추가

| 리스크 | 확률 | 영향 | 완화 방안 |
|--------|------|------|----------|
| 알림 피로 (너무 많은 알림) | 중 | 중 | 일일 알림 횟수 제한 (최대 3회) |
