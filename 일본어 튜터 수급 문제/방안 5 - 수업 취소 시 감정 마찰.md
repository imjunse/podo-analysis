# 방안 5: 수업 취소 시 감정 마찰 추가

> 상위 문서: [[문제 분석 및 해결 방안]]
> 난이도: 매우 낮음 | 소요: ~1주 | 기대 효과: 낮음 | 우선순위: Tier 3 - 소규모 테스트 후 결정

---

## 1. 핵심 가설

> "튜터가 수업을 취소할 때 학생과의 관계를 상기시키면, 취소 결정을 재고하는 비율이 높아진다."

**단, 이 가설에는 큰 단서가 붙는다:**
- JP 튜터에게 Podo는 부업
- 일본 문화에서 "감정적 부담"은 오히려 **플랫폼 자체를 회피**하게 만들 수 있음
- 취소를 줄이더라도 **근본적인 오픈 시간 부족은 해결 안 됨**

---

## 2. 현재 vs 변경 UX

### 2-1. 현재 (As-is)

```
[수업 취소]

"이 수업을 취소하시겠습니까?"

[취소하기]  [돌아가기]
```

### 2-2. 변경안 A: 소프트 마찰 (권장)

```
[수업 취소]

"[학생이름]님과의 수업입니다."
"이 학생은 선생님과 [N]번 수업을 진행했어요."

[취소하기]  [유지하기]
```

- 정보만 제공, 감정적 압박은 최소화
- 일본어: 「[학생이름]さんとのレッスンです。この生徒さんとは[N]回レッスンをされています。」

### 2-3. 변경안 B: 미디엄 마찰

```
[수업 취소]

"[학생이름]님과의 [N]번째 수업입니다."
"[학생이름]님이 남긴 메시지: '[최근 피드백]'"

[취소하기]  [유지하기]
```

- 피드백까지 보여줌으로써 관계를 더 구체화
- 일본어: 「[학생이름]さんからのメッセージ：「[피드백]」」

### 2-4. 변경안 C: 강한 마찰 (비권장)

```
"[학생이름]님이 선생님과의 수업을 기다리고 있어요.
정말 취소하시겠습니까?"
```

- **일본 문화에서는 이 수준의 마찰이 부담으로 작용할 확률 높음**
- 튜터가 "스케줄을 열면 취소가 부담스럽다" → 아예 스케줄을 덜 여는 역효과

---

## 3. 일본 문화 맥락 심층 분석

### 3-1. 왜 조심해야 하는가

| 한국 맥락 | 일본 맥락 |
|-----------|-----------|
| "학생이 기다리고 있어요" → 책임감 자극 → 긍정 반응 | "학생이 기다리고 있어요" → 부담/미안함 → 회피 |
| 감정 마찰 = 동기부여 도구 | 감정 마찰 = 심리적 부담 (프레셔) |
| 부업이라도 열심히 → 보상 | 부업에서까지 프레셔 → 플랫폼 이탈 |

일본의 "空気を読む(쿠우키오 요무)" 문화:
- 남에게 폐를 끼치는 것(迷惑をかける)을 극도로 꺼림
- "학생이 기다리고 있다"는 정보 = "취소하면 폐를 끼치는 것" → **스케줄을 처음부터 안 여는 것이 더 안전**이라는 결론에 도달할 수 있음

### 3-2. 안전한 접근 vs 위험한 접근

| 안전 (Do) | 위험 (Don't) |
|-----------|-------------|
| 사실 정보만 제공 (수업 횟수) | 감정에 호소 ("기다리고 있어요") |
| 선택지 제공 ("유지하기" 버튼) | 취소를 어렵게 만듦 (다단계 확인) |
| 튜터의 결정을 존중 | 죄책감 유발 |

---

## 4. A/B 테스트 설계

이 방안은 **반드시 소규모 테스트 후 결정**해야 함.

### 4-1. 테스트 구조

```
전체 JP 바이링구얼 튜터 (~290명)
    ├─ Control (50%): 현재 UX 유지
    └─ Treatment (50%): 변경안 A (소프트 마찰) 적용
```

### 4-2. 측정 지표

| 지표 | Control 예상 | 목표 | 위험 신호 |
|------|-------------|------|----------|
| 수업 취소율 | 기준값 측정 | -10% 이상 | 변화 없음 |
| 주간 스케줄 오픈 시간 | 8.5시간 | 유지 또는 증가 | **감소** (역효과) |
| 튜터 NPS / 만족도 | 기준값 측정 | 유지 | 하락 |
| 앱 사용 빈도 | 기준값 | 유지 | 감소 |

### 4-3. 역효과 감지 기준

**즉시 중단 조건:**
- Treatment 그룹의 주간 스케줄 오픈 시간이 Control 대비 **5% 이상 감소**
- Treatment 그룹에서 튜터 이탈(비활성화)이 Control 대비 유의미하게 증가

### 4-4. 테스트 기간

- **최소 2주** (취소 이벤트 충분히 수집)
- **권장 4주** (주간 변동 고려)

---

## 5. 기대 효과 시뮬레이션

### 5-1. 취소 건수 추정

- JP 튜터 290명 × 평균 8.5시간/주 × 60% 수업률 = ~주 1,480 수업
- 취소율이 5%라고 가정 → 주 ~74건 취소
- 10% 취소 감소 → 주 ~7.4건 수업 유지
- **월 ~30건 수업 유지 = 공급 면에서 매우 작은 효과**

### 5-2. 왜 기대 효과가 "낮음"인가

- 취소를 줄이는 것은 **기존 공급의 손실 방지**일 뿐
- 근본 문제는 **절대적 공급 부족** (290명 × 8.5시간이 총 수요를 커버하지 못함)
- 방안 2(보너스)가 +1,200 수업/주를 목표로 하는 것과 비교하면 미미

---

## 6. 구현 범위

### 단일 Phase (~3일)
- [ ] 수업 취소 확인 모달에 학생 정보 추가 (이름, 수업 횟수)
- [ ] A/B 테스트 플래그 설정 (50/50 분할)
- [ ] 취소 이벤트 로깅에 A/B 그룹 태그 추가
- [ ] 2주 후 데이터 리뷰 일정 설정

### 기술 스택
- 프론트엔드: 취소 모달 텍스트 변경 + API 1회 호출 (학생 정보)
- 백엔드: 기존 수업 이력 API에서 수업 횟수 반환 (이미 있을 가능성 높음)
- 테스트: 피처 플래그 (기존 A/B 테스트 인프라 활용)

---

## 7. 최종 권장

**조건부 진행:**
1. 변경안 A(소프트 마찰)로만 시작 — B, C는 하지 않음
2. 반드시 A/B 테스트로 진행 — 전체 적용 금지
3. 역효과 감지 기준 명확히 설정 후 시작
4. **2주 데이터 리뷰 후 결정** — 효과 있으면 유지, 역효과면 즉시 롤백

**기대 효과가 낮고 리스크가 있으므로, Tier 1 방안(웨이팅리스트, 보너스, 사전 공지)에 리소스를 우선 투입하고, 여유가 있을 때 이 테스트 진행을 권장.**
