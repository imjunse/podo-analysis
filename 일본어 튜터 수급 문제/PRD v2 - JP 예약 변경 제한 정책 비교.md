# PRD v2 — JP 예약 변경 제한 정책 비교

> sprint-kit scope: `jp-booking-change-limit-20260326-001`
> 작성일: 2026-03-27
> 상태: Draft
> 이전 버전: [[PRD - JP 예약 변경 횟수 제한]] (v1, 2026-03-26)

---

## 1. Executive Summary

일본어 수업이 풀부킹되어 학생들이 예약할 수 없고, 이로 인해 이탈(환불 33% + 낮은 리텐션)이 발생하고 있다. 분석 결과, **예약 변경이 신규 예약보다 많고**(131,553회 vs 119,208회), **355명의 슈퍼 체인저가 전체 변경의 61%**를 차지하고 있다.

v1에서는 "주 5회 제한"만 검토했으나, 팀 내 다른 의견이 제기되어 **3가지 정책 옵션을 ClickHouse 시뮬레이션으로 비교 분석**하고, 최적 정책을 선정한다.

> **중요: 이 정책이 해결하는 것과 해결하지 않는 것**
>
> 이 정책은 **슬롯 공급 자체를 늘리는 것이 아니다.** 고빈도 유저가 매일 1개 수업을 듣는 것은 동일하며, 슬롯 점유 총량은 변하지 않는다. 이 정책이 해결하는 것은 **슬롯이 계속 열렸다 닫히는 혼란**(일 1,450건 변경 → 슬롯 1,053개보다 많음)으로 인해 발생하는 "빈 자리인 줄 알았는데 이미 예약됨" 헛수고 경험이다.
>
> 실제 슬롯 공급 확대는 **튜터 인센티브 정책**(별도 프로젝트, 다른 팀원 담당)이 담당한다.

> **포지셔닝: 브릿지 솔루션**
>
> 이 정책은 풀부킹 문제의 **근본 해결책이 아니다.** 근본 해결은 **단계별 요금제**(Standard 주3회 / Premium 주5회 / Ultimate 주7회)로 수요를 가격으로 자연 분산하는 것이며, 이는 별도 프로젝트로 진행된다 ([[JP 수업 예약자 세그먼트 분석 및 가격 정책 제안]] §10 참조). 이 정책은 단계별 요금제 출시 전까지 **체감 풀부킹을 완화하는 브릿지 솔루션**이다.

### 핵심 목표

| 순위 | 목표 | 지표 |
|------|------|------|
| **1** | **풀부킹 체감 줄이기** | 일 풀부킹 알림 770회 → 400회 이하 |
| 부수 | 저빈도 유저 이탈 방지 | 취소율 8.6% → 5% 이하 |
| 부수 | 고빈도 충성 유저 만족도 유지 | 이탈 없이 변경 제한 수용 |

## 2. 문제 배경

### 핵심 수치 (2026년 1-3월)

| 지표 | 수치 |
|------|------|
| 3개월 총 예약 변경 (RESERVED→RESERVED) | **131,553회** |
| 3개월 신규 예약 (CREATED→RESERVED) | 119,208회 |
| 변경/신규 비율 | **1.1배** (변경이 더 많음) |
| 일평균 예약 변경 | **~1,450건** |
| 일 전체 슬롯 수 | ~1,053개 |
| 변경/슬롯 비율 | **1.4배** (슬롯보다 변경이 많음) |

### 슈퍼 체인저 현황

| 그룹 | 유저 수 | 총 변경 건수 | 평균 변경 | 수강권 |
|------|---------|------------|----------|--------|
| 51회+ (슈퍼 체인저) | **355명** | **80,734건 (61%)** | 227.4회 | UNLIMIT 313명 (89%) |
| 21-50회 | 781명 | 24,290건 | 31.1회 | - |
| 6-20회 | 1,766명 | 19,489건 | 11.0회 | - |
| 1-5회 | 3,227명 | 7,040건 | 2.2회 | - |

### 변경이 만드는 "가짜 풀부킹" 현상

슈퍼 체인저가 하루에 여러 번 시간/튜터를 바꾸면, 인기 시간대 슬롯이 반복적으로 점유↔해제를 오간다. 이 과정에서 다른 학생이 "빈 자리"를 보고 클릭하면 이미 다시 점유된 상태 → **풀부킹 알림 발생**. 고빈도 유저가 최종적으로 1개 수업만 듣더라도, 변경 과정 자체가 다른 학생의 예약 경험을 방해한다.

```
슈퍼 체인저 A의 하루 (실제 수업은 1개):

09:00  21시 슬롯 예약 (21시 점유)
10:00  21시→20시 변경 (21시 해제→20시 점유) ← 21시 잠깐 빈 자리로 보임
12:00  20시→22시 변경 (20시 해제→22시 점유) ← 20시 잠깐 빈 자리로 보임
14:00  22시→21시 변경 (22시 해제→21시 점유) ← 22시 잠깐 빈 자리로 보임

→ 수업 1개, 하지만 슬롯 4개가 영향 받음
→ 이걸 355명이 하면 일 1,450건 변경 (슬롯 1,053개보다 많음)
```

### 변경 밀도와 풀부킹 알림의 용량-반응 관계

ClickHouse 분석 결과, **10분 구간별 변경 밀도가 높을수록 풀부킹 알림이 기하급수적으로 증가**:

| 10분 동안 변경 건수 | 평균 풀부킹 알림 | 배율 | 해석 |
|---|---|---|---|
| 1-2건 (한산) | 0.57 | 기준 | ≈ 진짜 풀부킹 수준 |
| 3-5건 | 1.34 | 2.4배 | 변경 영향 시작 |
| 6-10건 | 3.57 | **6.3배** | 변경 영향 뚜렷 |
| **11건+** (폭주) | **14.21** | **25배** | 변경이 알림을 지배 |

> 변경 밀도가 가장 높은 구간(11건+)에서 **전체 풀부킹 알림의 76.5%** (15,629/20,423건)가 발생. 이 구간에서 진짜 풀부킹은 ~4% (0.57/14.21)에 불과하고, 나머지 ~96%는 변경으로 인한 슬롯 불안정이 만들어낸 체감 풀부킹으로 추정.

## 3. 정책 옵션 비교 분석

### 3.1 후보 옵션

| 옵션 | 설명 | 제한 단위 |
|------|------|----------|
| **A. 주 5회 제한** | UNLIMIT 유저의 주간 변경 총량을 5회로 제한. 매주 월요일 리셋 | 유저-주 |
| **B. 당일 변경 불가** | 캘린더 날짜 기준 당일 수업은 변경 불가 | 변경 시점 |
| **C-1. 수업당 1회** | 개별 수업(예약) 단위 변경 1회만 가능 | 수업 |
| **C-2. 수업당 2회** | 개별 수업(예약) 단위 변경 2회까지 가능 | 수업 |

### 3.2 공통 규칙 (모든 옵션 동일)

| 항목 | 규칙 |
|------|------|
| 적용 대상 | UNLIMIT 수강권 (originCount == 999)만. COUNT/TRIAL 미적용 |
| 변경 정의 | 슬롯(시간) 변경 = 변경 1회. 튜터는 랜덤 매칭이므로 별도 변경 불가 |
| 취소→재예약 | 변경 1회로 카운트 (우회 방지) |
| 튜터 취소 | 카운트 비대상 (유저 귀책 아님) |
| 어드민 변경 | 카운트 비대상 |
| 제한 도달 시 | [변경] 버튼 클릭 시 팝업으로 안내. 기존 예약은 유지 |
| 피처 플래그 | 단계적 롤아웃 + 긴급 시 즉시 off |

### 3.2.1 UI 원칙: 최소 변화

> **핵심: 백엔드 로직 변경이 메인, 프론트 UI 변화는 최소화**

| 원칙 | 상세 |
|------|------|
| **기존 화면 유지** | 예약 상세 화면의 전체 레이아웃/디자인 변경 없음 |
| **변경 버튼 옆 남은 횟수** | [변경] CTA 옆에 `(2/5)` 같은 최소한의 남은 횟수만 표시. 그 외 화면 변화 없음 |
| **최초 1회 공지 팝업** | 정책 도입 시 예약 화면에서 모든 UNLIMIT 유저에게 1회만 팝업. 닫으면 다시 표시 안 함 |
| **제한 도달 시 팝업 안내** | [변경] 버튼을 눌렀을 때, 제한에 걸린 경우에만 팝업으로 "변경 횟수를 모두 사용했습니다" 안내 |

이렇게 하면:
- 유저가 느끼는 디자인 변화는 **변경 버튼 옆 숫자 1개**뿐
- 제한에 걸린 유저만 팝업으로 인지
- 프론트엔드 개발 범위가 **공지 팝업 1개 + 제한 팝업 1개 + 버튼 옆 카운터**로 최소화

### 3.6 추천

**풀부킹 감소가 최우선 목표**이므로, 두 가지 유력 후보:

1. **C-1 (수업당 1회)** — 차단율 70%로 최대 효과. 단, 유저 불만/이탈 리스크가 큼. 피처 플래그로 소규모 롤아웃 후 이탈 모니터링 필수.
2. **A (주 5회)** — 차단율 59.3%로 준수한 효과 + 우회 방어가 가장 강력. 유저 자기 통제감 제공.

> **판단 포인트**: "70% 차단이지만 46% 수업이 영향받는 C-1" vs "59% 차단이지만 80% 유저-주가 영향 없는 A". 이탈 리스크를 얼마나 감수할 수 있느냐에 따라 결정.

## 4. 선택된 정책의 상세 규칙

> 아래는 **각 옵션이 선택될 경우**의 규칙 상세. 옵션 결정 후 해당 섹션만 유지.

### 4-A. 주 5회 제한 규칙

#### 카운트 규칙

```
카운트 대상:
  (1) 예약 변경 (RESERVED → RESERVED)                   = 1회 차감
  (2) 취소 후 동일 주(월~일) 내 재예약                    = 1회 차감

카운트 비대상:
  (1) 튜터 취소로 인한 강제 변경               = 차감 안 함
  (2) 어드민(changeByAdmin)에 의한 변경         = 차감 안 함
  (3) 취소만 하고 재예약 안 함 (진짜 취소)      = 차감 안 함
  (4) 다음 주 이후 재예약 (리셋 후)            = 새 주의 카운터로 계산
```

| 항목 | 규칙 |
|------|------|
| 대상 수강권 | UNLIMIT (originCount == 999) |
| 주간 한도 | **5회** |
| 리셋 시점 | 매주 **월요일 00:00 KST** |
| 제한 도달 시 | 변경 버튼 비활성 + 안내 메시지 |
| 기존 정책 유지 | 수업 시작 2시간 전 변경 마감 (독립) |
| 정책 시행 첫 주 | 0/5에서 시작 (소급 적용 안 함) |

#### 우회 방지: 취소→재예약 감지

> "동일 주(월~일) 내" 취소 후 재예약을 1회로 카운트.
> 감지율: 86% (24시간 룰 대비 +22%p 향상)

#### 기존 취소 페널티와의 상호작용

| 취소 시점 | 기존 페널티 | 우회 가능 여부 |
|----------|-----------|--------------|
| 수업 2시간+ 전 | 없음 | **동일 주 내 재예약 감지로 차단** |
| 수업 1~2시간 전 | 72시간 예약 금지 | **불가** — hold로 재예약 자체 안 됨 |
| 수업 1시간 이내 | 72시간 + 환불 | **불가** |

### 4-B. 당일 변경 불가 규칙

```
제한 규칙:
  캘린더 날짜 기준, 오늘(KST) 날짜의 수업은 변경 불가.
  예: 3/27 수업은 3/27에 변경 불가. 3/26까지는 자유.

비대상:
  (1) 튜터 취소로 인한 강제 변경 = 제한 안 함
  (2) 어드민 변경 = 제한 안 함
```

| 항목 | 규칙 |
|------|------|
| 대상 수강권 | UNLIMIT (originCount == 999) |
| 제한 기준 | 변경 시점(KST)의 날짜 = 수업 날짜이면 차단 |
| 전일 변경 | **자유** (제한 없음) |
| 기존 정책 유지 | 수업 시작 2시간 전 변경 마감 (독립) |

### 4-C. 수업당 변경 제한 규칙

```
카운트 대상:
  개별 수업(class_id) 단위로 변경 횟수를 추적.
  수업 A를 1회 변경하면 A의 카운터 = 1.
  수업 B를 2회 변경하면 B의 카운터 = 2.
  각 수업은 독립적.

카운트 비대상:
  (1) 튜터 취소로 인한 강제 변경 = 차감 안 함
  (2) 어드민 변경 = 차감 안 함
```

| 항목 | C-1 (1회) | C-2 (2회) |
|------|----------|----------|
| 대상 수강권 | UNLIMIT | UNLIMIT |
| 수업당 한도 | **1회** | **2회** |
| 제한 도달 시 | 해당 수업의 변경 버튼 비활성 | 해당 수업의 변경 버튼 비활성 |
| 다른 수업 | **영향 없음** (수업별 독립) | **영향 없음** |
| 리셋 | 없음 (수업 단위이므로 리셋 개념 없음) | 없음 |
| 기존 필드 활용 | `CLASS_CHANGE_SEQ` | `CLASS_CHANGE_SEQ` |

#### 우회 방지: 취소→재예약

수업을 취소하고 새로 예약하면 새 class_id가 생성되므로, **변경 카운터가 0으로 리셋**됨.

방어 방법:
- 취소 후 동일 날짜/시간대 재예약을 감지하여 이전 수업의 변경 카운터를 승계
- 또는 취소 후 일정 시간 내 재예약 시 카운터 1회 추가 차감

> ⚠ 이 우회 방어는 옵션 A(주 5회)의 "동일 주 내 감지"보다 구현이 복잡하고, 감지율도 낮을 수 있음.

## 5. User Journeys

### Journey 1: 정상 변경 (Happy Path)

**Persona:** 유미 (28세, 직장인, UNLIMIT 구독 3개월차, 주 3-4회 수업)

1. 예약 상세 화면 진입 — **기존과 동일한 화면**. [변경] 버튼 옆에 `(1/5)` 작은 카운터만 표시
2. [변경] 클릭 → 기존과 동일하게 예약 화면으로 이동
3. 변경 완료 → 카운터 `(2/5)`로 업데이트
4. 유미는 카운터를 신경쓰지 않아도 됨 — **체감 변화 없음**

### Journey 2: 변경 소진 (Exception Path)

**Persona:** 다이치 (32세, 프리랜서, UNLIMIT 구독 6개월차, 매일 수업, 슈퍼 체인저)

1. 예약 상세 화면 — [변경] 버튼 옆 `(5/5)` 표시
2. [변경] 클릭 → **팝업 표시**: "이번 주 변경 횟수를 모두 사용했습니다. N일 후(월요일) 초기화됩니다."
3. 팝업 닫기 → 기존 예약은 그대로 유지
4. 다이치는 수업을 그대로 듣기로 결정

> **UI 포인트**: 변경 버튼 자체는 항상 활성 상태. 클릭 후 팝업으로만 안내.

### Journey 3: 취소→재예약 우회 시도

| 옵션 A | 옵션 B | 옵션 C |
|--------|--------|--------|
| 동일 주 내 감지 → 차단 | 당일이면 재예약도 차단? (규칙 모호) | **새 class_id → 카운터 리셋** (우회 가능) |
| 감지율 86% | 감지 불필요 (날짜 체크만) | 추가 감지 로직 필요 |

### Journey 4: 튜터 취소로 인한 강제 변경

모든 옵션 동일: **카운터 미증가**, 유저 불이익 없음.

### Journey 5: 정책 최초 도입 공지

1. 정책 도입 후 첫 접속 시, 예약 화면에서 **공지 팝업 1회** 표시
2. [확인] 클릭 → 팝업 닫힘 (다시 표시 안 됨)
3. 이후 화면은 **기존과 동일** — 변경 버튼 옆 카운터만 추가

| 옵션 | 공지 팝업 메시지 |
|------|---------------|
| A | "더 많은 학생이 원하는 시간에 수업할 수 있도록, 주간 예약 변경 횟수가 5회로 제한됩니다 (매주 월요일 초기화)" |
| B | "더 많은 학생이 원하는 시간에 수업할 수 있도록, 당일 수업의 예약 변경이 제한됩니다" |
| C | "더 많은 학생이 원하는 시간에 수업할 수 있도록, 예약 변경은 수업당 N회까지 가능합니다" |

## 12. 리스크 및 완화 방안

| 리스크 | 심각도 | 완화 방안 |
|--------|--------|----------|
| 슈퍼 체인저 이탈 (해지) | 높음 | 피처 플래그로 단계적 롤아웃. 해지율 급증 시 즉시 off. 슈퍼 체인저 296명 누적 결제 3.17억원 (전체의 4.4%) |
| CS 문의 급증 | 중간 | 정책 공지 팝업 + 변경 화면 상시 안내. CS 매뉴얼 준비 |
| 취소→재예약 우회 | 중간 | 옵션별 우회 방어 (A: 동일 주 감지 / B: 불필요 / C: 카운터 승계) |
| 변경 감소가 풀부킹에 미치는 효과 불확실 | 중간 | 2주간 집중 모니터링, KPI 미달 시 정책 강화/완화 조정 |

### 12.1 롤백 기준 (Kill Criteria)

피처 플래그 off 판단 기준. 아래 중 **하나라도** 충족 시 즉시 롤백:

| 지표 | 롤백 기준 | 측정 방법 | 판단 주기 |
|------|----------|----------|----------|
| UNLIMIT 해지율 | 롤아웃 전 대비 **+20% 이상** 증가 | UNLIMIT 주간 해지 건수 | 일별 |
| CS 문의 건수 | 예약 변경 관련 CS **일 50건** 이상 | CS 티켓 태그 분류 | 일별 |
| 풀부킹 알림 | 오히려 **증가** (770회 → 800회 이상 지속) | 이벤트 집계 | 1주 관찰 |
| 고빈도 유저 NPS | 만족도 급락 시 | 인앱 서베이 | 출시 2주 후 |

> **롤백 프로세스**: 피처 플래그 off → 기존 동작 복원 → 원인 분석 → 수치 조정 후 재롤아웃

### CS 대응 매뉴얼

| 문의 유형 | 응대 가이드 |
|----------|-----------|
| "왜 변경이 안 되나요?" | [옵션별 설명] + "더 많은 학생이 원하는 시간에 수업할 수 있도록 정책이 변경되었습니다" |
| "무제한인데 제한하면 어떡해요" | "수업 횟수는 무제한 그대로입니다. 예약 변경만 제한됩니다." |
| "급한 변경인데 안 돼요" | "튜터 사정으로 인한 변경은 제한 없습니다. 불가피한 경우 CS를 통해 어드민 변경을 요청해주세요." |
| "환불하고 싶어요" | 이탈 방지 대응 → 담당자 에스컬레이션 |

### 슈퍼 체인저 특별 대응

- 도입 초기 2주간, 변경 한도 도달 유저 중 CS 문의 or 해지 시도 유저를 별도 리스트로 관리
- 필요 시 1:1 안내 또는 한시적 한도 상향 검토 (피처 플래그 유저 그룹 분리)
