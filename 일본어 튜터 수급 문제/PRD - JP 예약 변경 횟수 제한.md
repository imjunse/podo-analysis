# PRD — JP 예약 변경 횟수 제한

> sprint-kit scope: `jp-booking-change-limit-20260326-001`
> 작성일: 2026-03-26
> 상태: Draft

---

## 1. Executive Summary

일본어 수업이 풀부킹되어 학생들이 예약할 수 없고, 이로 인해 이탈(환불 33% + 낮은 리텐션)이 발생하고 있다. 분석 결과, **예약 변경이 신규 예약보다 많고**(131,553회 vs 119,208회), **355명의 슈퍼 체인저가 전체 변경의 61%**를 차지하고 있다.

UNLIMIT(무제한 수강권) 유저 대상으로 **주간 예약 변경 총량을 5회로 제한**하여, 잦은 변경으로 인한 슬롯 변동(가짜 풀부킹)을 줄이고 예약 안정성을 높인다.

> **중요: 이 정책이 해결하는 것과 해결하지 않는 것**
>
> 이 정책은 **슬롯 공급 자체를 늘리는 것이 아니다.** 고빈도 유저가 매일 1개 수업을 듣는 것은 동일하며, 슬롯 점유 총량은 변하지 않는다. 이 정책이 해결하는 것은 **슬롯이 계속 열렸다 닫히는 혼란**(일 1,450건 변경 → 슬롯 1,053개보다 많음)으로 인해 발생하는 "빈 자리인 줄 알았는데 이미 예약됨" 헛수고 경험이다.
>
> 실제 슬롯 공급 확대는 **튜터 인센티브 정책**(별도 프로젝트, 다른 팀원 담당)이 담당한다.

> **포지셔닝: 브릿지 솔루션**
>
> 이 정책은 풀부킹 문제의 **근본 해결책이 아니다.** 근본 해결은 **단계별 요금제**(Standard 주3회 / Premium 주5회 / Ultimate 주7회)로 수요를 가격으로 자연 분산하는 것이며, 이는 별도 프로젝트로 진행된다 ([[JP 수업 예약자 세그먼트 분석 및 가격 정책 제안]] §10 참조). 이 정책은 단계별 요금제 출시 전까지 **체감 풀부킹을 완화하는 브릿지 솔루션**이다.

### 핵심 목표

| 순위 | 목표 | 지표 |
|------|------|------|
| **1** | **풀부킹 체감 줄이기** | 일 풀부킹 알림 770회 → 400회 이하 |
| 부수 | 저빈도 유저 이탈 방지 | 취소율 8.6% → 5% 이하 |
| 부수 | 고빈도 충성 유저 만족도 유지 | 이탈 없이 변경 제한 수용 |

## 2. 문제 배경

### 핵심 수치 (2026년 1-3월)

| 지표 | 수치 |
|------|------|
| 3개월 총 예약 변경 (RESERVED→RESERVED) | **131,553회** |
| 3개월 신규 예약 (CREATED→RESERVED) | 119,208회 |
| 변경/신규 비율 | **1.1배** (변경이 더 많음) |
| 일평균 예약 변경 | **~1,450건** |
| 일 전체 슬롯 수 | ~1,053개 |
| 변경/슬롯 비율 | **1.4배** (슬롯보다 변경이 많음) |

### 슈퍼 체인저 현황

| 그룹 | 유저 수 | 총 변경 건수 | 평균 변경 | 수강권 |
|------|---------|------------|----------|--------|
| 51회+ (슈퍼 체인저) | **355명** | **80,734건 (61%)** | 227.4회 | UNLIMIT 313명 (89%) |
| 21-50회 | 781명 | 24,290건 | 31.1회 | - |
| 6-20회 | 1,766명 | 19,489건 | 11.0회 | - |
| 1-5회 | 3,227명 | 7,040건 | 2.2회 | - |

### 변경이 만드는 "가짜 풀부킹" 현상

슈퍼 체인저가 하루에 여러 번 시간/튜터를 바꾸면, 인기 시간대 슬롯이 반복적으로 점유↔해제를 오간다. 이 과정에서 다른 학생이 "빈 자리"를 보고 클릭하면 이미 다시 점유된 상태 → **풀부킹 알림 발생**. 고빈도 유저가 최종적으로 1개 수업만 듣더라도, 변경 과정 자체가 다른 학생의 예약 경험을 방해한다.

```
슈퍼 체인저 A의 하루 (실제 수업은 1개):

09:00  21시 슬롯 예약 (21시 점유)
10:00  21시→20시 변경 (21시 해제→20시 점유) ← 21시 잠깐 빈 자리로 보임
12:00  20시→22시 변경 (20시 해제→22시 점유) ← 20시 잠깐 빈 자리로 보임
14:00  22시→21시 변경 (22시 해제→21시 점유) ← 22시 잠깐 빈 자리로 보임

→ 수업 1개, 하지만 슬롯 4개가 영향 받음
→ 이걸 355명이 하면 일 1,450건 변경 (슬롯 1,053개보다 많음)
```

### 풀부킹 체감 피해

| 지표 | 수치 |
|------|------|
| 일평균 풀부킹 알림 | ~770회 |
| 일평균 영향받는 유저 | ~82명 |
| 저빈도 유저 취소율 | 8.6% (고빈도의 3배) |
| 3월 booking 관련 JP 환불 | 498만원 (월 기준 최고치 페이스) |

### 변경 밀도와 풀부킹 알림의 용량-반응 관계

ClickHouse 분석 결과, **10분 구간별 변경 밀도가 높을수록 풀부킹 알림이 기하급수적으로 증가**:

| 10분 동안 변경 건수 | 평균 풀부킹 알림 | 배율 | 해석 |
|---|---|---|---|
| 1-2건 (한산) | 0.57 | 기준 | ≈ 진짜 풀부킹 수준 |
| 3-5건 | 1.34 | 2.4배 | 변경 영향 시작 |
| 6-10건 | 3.57 | **6.3배** | 변경 영향 뚜렷 |
| **11건+** (폭주) | **14.21** | **25배** | 변경이 알림을 지배 |

> 변경 밀도가 가장 높은 구간(11건+)에서 **전체 풀부킹 알림의 76.5%** (15,629/20,423건)가 발생. 이 구간에서 진짜 풀부킹은 ~4% (0.57/14.21)에 불과하고, 나머지 ~96%는 변경으로 인한 슬롯 불안정이 만들어낸 체감 풀부킹으로 추정.
>
> 데이터 소스: `podo_events.events` + `le_class_status_history`, 2026-03-01~25, 상세 §부록 B-1

### 이 정책이 해결하는 것 vs 해결하지 않는 것

| 해결하는 것 (체감 개선) | 해결하지 않는 것 (공급 문제) |
|----------------------|--------------------------|
| 슬롯이 반복적으로 열렸다 닫히는 혼란 | 슬롯 절대 수 부족 |
| "빈 자리인 줄 알았는데 이미 예약됨" 헛수고 | 고빈도 유저의 슬롯 점유 총량 |
| 풀부킹 알림 770회 → 감소 (체감 개선) | 실제 가용 슬롯 수 증가 |

## 4. 정책 규칙 상세

### 4.1 변경 카운트 규칙

```
카운트 대상:
  (1) 예약 변경 (RESERVED → RESERVED)                   = 1회 차감
  (2) 취소 후 동일 주 내 재예약 (CANCEL → 신규 RESERVED) = 1회 차감
      ※ "동일 주" = 같은 월요일~일요일 구간 (리셋 주기와 동일)

카운트 비대상:
  (1) 튜터 취소로 인한 강제 변경               = 차감 안 함
  (2) 어드민(changeByAdmin)에 의한 변경         = 차감 안 함
  (3) 취소만 하고 재예약 안 함 (진짜 취소)      = 차감 안 함
  (4) 다음 주 이후 재예약 (리셋 후)            = 새 주의 카운터로 계산
```

> **24시간 → 동일 주 내로 변경한 이유**
>
> ClickHouse 분석 결과, 취소 후 재예약의 시간 간격 분포:
> - 1시간 이내: 41.8% (이미 "변경 대용"으로 사용 중)
> - 1-24시간: 22.3%
> - **24시간-1주: 21.9%** ← 24시간 룰에서는 우회됨
>
> 고빈도 유저 98.6%가 당일/전날 예약이므로, 24시간만 기다리면 사실상 우회 가능.
> "동일 주 내" 룰로 변경하면 감지율 64% → **86%**로 상승.
> 취소만 하고 재예약 안 하는 "진짜 취소"는 카운트되지 않으므로 부작용 없음.

### 4.1.1 기존 취소 페널티와의 상호작용

취소→재예약 우회 시도 시, 기존 UNLIMIT 취소 페널티(72시간 예약 금지)가 자연적 방어선 역할을 한다:

| 취소 시점 | 기존 페널티 | 변경 제한 우회 가능 여부 |
|----------|-----------|---------------------|
| 수업 2시간+ 전 | 없음 (CANCEL + 수업권 복구) | **동일 주 내 재예약 감지로 차단** |
| 수업 1~2시간 전 | **72시간 예약 금지** | **불가** — hold 때문에 재예약 자체가 안 됨 |
| 수업 1시간 이내 | **72시간 예약 금지** + 환불 | **불가** — hold 때문에 재예약 자체가 안 됨 |

> early cancel(2시간+ 전)은 **동일 주 내 재예약 감지**로 차단, late cancel은 기존 **72시간 hold**가 자연 차단. 두 방어선이 겹치지 않는 영역 없이 모든 우회 시나리오를 커버.
>
> ⚠ **참고**: 72시간 hold 정책은 현재 별도 담당자가 완화를 검토 중. hold 정책이 변경되면 이 상호작용도 재검토 필요.

### 4.2 제한 조건

| 항목 | 규칙 |
|------|------|
| 대상 수강권 | UNLIMIT (originCount == 999) |
| 주간 한도 | **5회** |
| 리셋 시점 | 매주 **월요일 00:00 KST** (서비스 전체 기준 시간대와 동일) |
| 제한 도달 시 | 변경 버튼 비활성 + 안내 메시지 |
| 기존 정책 유지 | 수업 시작 2시간 전 변경 마감 (변경 제한과 독립) |
| 정책 시행 첫 주 | **0/5에서 시작** (시행 전 변경 횟수 소급 적용 안 함) |
| 일요일/월요일 경계 | 일요일 밤 5회 + 월요일 리셋 후 5회 = 이틀 사이 최대 10회 가능. v1에서는 수용 (주 단위 캡의 구조적 특성, 실제 발생 빈도 낮을 것으로 예상) |

### 4.3 수강권 변경/업그레이드 시

| 상황 | 처리 |
|------|------|
| COUNT → UNLIMIT 업그레이드 | 업그레이드 주는 제한 없음, 다음 월요일부터 적용 |
| UNLIMIT → UNLIMIT 연장 | 카운트 유지 (연속) |
| UNLIMIT 만료 후 재구매 | 신규 시작, 0/5로 리셋 |

## 5. User Journeys

### Journey 1: 정상 변경 (Happy Path)

**Persona:** 유미 (28세, 직장인, UNLIMIT 구독 3개월차, 주 3-4회 수업)

**Opening Scene:** 유미는 내일 21시에 예약한 수업을 22시로 옮기고 싶다. 예약 상세 화면에 들어간다.

**Rising Action:**
1. 예약 상세 화면에서 **`변경 (1/5)`** 표시를 확인 — 이번 주 1회 사용, 4회 남음
2. [변경] 버튼 클릭 → 예약 화면으로 이동
3. 22시, 박 튜터 선택 → 변경 확인

**Climax:** 변경 완료. 기존 21시 슬롯 해제, 22시 슬롯 점유.

**Resolution:** 예약 상세로 돌아오면 **`변경 (2/5)`**로 업데이트. 유미는 "아직 3번 더 바꿀 수 있구나" 확인.

---

### Journey 2: 변경 소진 (Exception Path)

**Persona:** 다이치 (32세, 프리랜서, UNLIMIT 구독 6개월차, 매일 수업, 슈퍼 체인저)

**Opening Scene:** 다이치는 이번 주에 이미 5번 변경을 사용했다. 오늘 예약한 수업의 튜터를 바꾸고 싶다.

**Rising Action:**
1. 예약 상세 화면에서 **`변경 (5/5)`** 표시 확인
2. [변경] 버튼이 **비활성** 상태
3. 안내 메시지: **"이번 주 변경 횟수를 모두 사용했습니다. 3일 후(월요일) 초기화됩니다."**

**Climax:** 다이치는 변경할 수 없다는 것을 확인.

**Resolution:** 다이치는 수업을 그대로 듣기로 결정. "다음 주에는 변경을 아껴서 써야겠다."

---

### Journey 3: 취소→재예약 우회 시도 (Exception Path)

**Persona:** 하루 (25세, 대학원생, UNLIMIT 구독 2개월차, 매일 수업)

**Opening Scene:** 하루는 변경 5/5를 다 썼지만, 내일 수업 튜터를 바꾸고 싶다. "취소하고 다시 예약하면 되지 않을까?" 생각한다.

**Rising Action:**
1. 수업 취소 (수업 시작 3시간 전이라 페널티 없음)
2. 다음 날, 새로운 시간/튜터로 예약 시도 ("24시간 넘겼으니 되겠지?")

**Climax:** 시스템이 **같은 주(월~일) 내 취소→재예약**을 감지 → "이번 주 변경 횟수를 모두 사용했습니다. 3일 후(월요일) 초기화됩니다." 안내.

**Resolution:** 하루는 재예약이 차단되는 것을 확인. "24시간 기다려도 안 되네. 같은 주 안에는 변경이든 취소→재예약이든 같구나."

> **주의**: 이 시나리오에서 취소 자체는 이미 완료된 상태. 재예약만 차단됨. 기존 슬롯은 이미 해제되었으므로, 학생은 수업을 잃게 됨. 이 리스크는 "소진 시 안내 UI"에서 **취소 전에** 충분히 경고해야 함. (예: "변경 횟수를 모두 사용했습니다. 취소 후 이번 주에는 재예약이 제한됩니다.")

### Journey 4: 튜터 취소로 인한 강제 변경 (Exception Path)

**Persona:** 민지 (30세, 직장인, UNLIMIT, 이번 주 변경 4/5)

**Opening Scene:** 민지의 내일 수업 튜터가 갑자기 수업을 취소했다. 시스템이 자동으로 다른 시간/튜터를 배정하거나, 민지에게 재선택을 요청한다.

**Rising Action:**
1. 튜터 취소 알림 수신
2. 다른 시간/튜터로 변경

**Climax:** 변경 완료. **변경 카운트는 여전히 4/5** (차감되지 않음).

**Resolution:** 민지는 "튜터 사정인데 내 횟수가 깎이면 억울했을 텐데, 다행이다."

## 9. 효과 측정 KPI

| KPI | 현재 | 목표 | 측정 방법 | **데이터 근거** |
|-----|------|------|----------|--------------|
| 일 예약 변경 횟수 | ~1,450건 | 700건 이하 | 백엔드 로그 | **ClickHouse 시뮬레이션: 주 5회 캡으로 59.3% 감소 → ~679건 예상** (§부록 B-2) |

| 취소→재예약 우회 시도 | 측정 필요 | 모니터링 | 취소 후 동일 주 내 재예약 건수 |

### "주 5회" 기준의 세그먼트별 영향 시뮬레이션

UNLIMIT 유저의 수업 빈도 분석([[JP 수업 예약자 세그먼트 분석 및 가격 정책 제안]] §3.2)에 근거한 예상 영향:

| UNLIMIT 세그먼트 | 유저 수 | 주당 수업 | 변경 제한 영향 |
|-----------------|---------|----------|-------------|
| 저빈도 | 493명 | ~0.8회 | ✅ **영향 없음** |
| 중빈도 | 607명 | ~2.7회 | ✅ **영향 없음** |
| 고빈도A (13-30회) | 763명 | ~6.8회 | ⚠ **경미** — 수업 횟수 > 변경 한도이나, 대부분 1:1 비율 아님 |
| 고빈도B (31-60회) | 615명 | ~14.5회 | 🔴 **크게 제한** — 주 5회 변경으로는 일부 수업만 변경 가능 |
| 초고빈도 (61회+) | 259명 | ~23.8회 | 🔴 **가장 크게 제한** — 슈퍼 체인저 핵심 대상 |

> **주 5회 기준 선정 근거**: UNLIMIT 유저의 68% (저빈도+중빈도+고빈도A = 1,863명)에게 사실상 영향 없이, 슈퍼 체인저(고빈도B+초고빈도 = 874명)의 과도한 변경만 제한. 주 5회는 "매일 수업을 듣되, 필요 시 하루 1번은 변경 가능"한 수준.
>
> ✅ **검증 완료**: 실제 변경 횟수 기준으로도 ClickHouse 시뮬레이션 결과, 주 5회 캡으로 **전체 변경의 59.3%가 차단**되며, **80%의 유저-주는 영향 없음** 확인 (→ §부록 B-2).

### 모니터링 대시보드 (출시 후 2주간 집중)

- 일별 변경 횟수 추이
- 변경 제한 도달 유저 수
- 변경 제한 도달 후 취소→재예약 시도 건수
- UNLIMIT 해지 추이 (이탈 조기 감지)

## 12. 리스크 및 완화 방안

| 리스크 | 심각도 | 완화 방안 |
|--------|--------|----------|
| 슈퍼 체인저 이탈 (해지) | 높음 | 피처 플래그로 단계적 롤아웃. 해지율 급증 시 즉시 off. **슈퍼 체인저 296명의 누적 결제 총액 3.17억원 (전체 UNLIMIT JP 매출의 4.4%), 인당 평균 결제 ~107만원 (일반의 1.6배)**. 단, 변경 제한으로 해방되는 슬롯 가치 + 풀부킹 환불 방지(월 498만원+α)를 고려하면 정당화 가능 |

| 변경 감소가 풀부킹에 미치는 효과 불확실 | 중간 | 2주간 집중 모니터링, KPI 미달 시 주 3회로 조정 검토. **가짜 풀부킹 vs 진짜 풀부킹 비율 사전 분석 권장** (→ §부록 B) |

### 12.1 롤백 기준 (Kill Criteria)

피처 플래그 off 판단 기준. 아래 중 **하나라도** 충족 시 즉시 롤백:

| 지표 | 롤백 기준 | 측정 방법 | 판단 주기 |
|------|----------|----------|----------|
| UNLIMIT 해지율 | 롤아웃 전 대비 **+20% 이상** 증가 | UNLIMIT 주간 해지 건수 | 일별 모니터링 |
| CS 문의 건수 | 예약 변경 관련 CS가 **일 50건** 이상 | CS 티켓 태그 분류 | 일별 모니터링 |
| 풀부킹 알림 | 오히려 **증가** (770회 → 800회 이상 지속) | `booking_alert_too-many-requests` | 출시 후 1주 관찰 |
| 고빈도 유저 NPS | 별도 서베이에서 **만족도 급락** 시 | 인앱 서베이 (선택) | 출시 2주 후 |

> **롤백 프로세스**: 피처 플래그 off → 기존 동작 복원 → 원인 분석 → 주 5회 외 다른 수치(주 7회, 주 10회 등) 재검토 후 재롤아웃

## 부록 B: PRD 확정 전 추가 검증 필요 항목

아래 항목은 PRD의 논리적 근거를 강화하기 위해 ClickHouse 쿼리로 추가 분석이 필요한 항목이다.

### B-1. 풀부킹 알림과 예약 변경의 상관관계 ✅ 검증 완료

> 데이터 소스: `podo_events.events` (popup_viewed → booking_alert_too-many-requests) + `podo_mysql.le_class_status_history` (RESERVED→RESERVED)
> 분석 기간: 2026-03-01 ~ 2026-03-25

#### 시간대별 상관관계

| 시간대 | 일평균 풀부킹 알림 | 일평균 변경 | 상관계수 |
|--------|------------------|-----------|---------|
| 0시 | 60.3 | 78.3 | **0.611** |
| 5시 | 22.4 | 16.7 | **0.756** |
| 8시 | 30.5 | 42.1 | **0.788** |
| 16시 | 29.2 | 54.3 | **0.685** |
| 19시 | 66.1 | 75.9 | **0.685** |
| 21시 | 41.5 | 81.9 | 0.568 |
| 22시 | 71.6 | 88.3 | 0.330 |
| 23시 | 83.5 | 109.4 | **0.661** |

> 대부분의 시간대에서 **풀부킹 알림과 변경 건수 간 0.3~0.8의 양의 상관관계** 확인. 특히 새벽(0시), 오전(8시), 저녁(19시) 시간대에서 상관관계가 높음.
>
> **해석**: 변경이 많은 시간대에 풀부킹 알림도 많이 발생. 이는 변경으로 인한 슬롯 불안정이 풀부킹 체감에 기여한다는 가설을 **지지**함. 단, 상관관계 ≠ 인과관계이므로, 실제 정책 적용 후 A/B 비교가 필요.

#### 일별 변경/알림 비율

| 날짜 유형 | 일평균 알림 | 일평균 변경 | 변경/알림 비율 |
|----------|-----------|-----------|-------------|
| 일반일 (평균) | ~770 | ~1,250 | 1.5~2.0 |
| 피크일 (3/8) | 1,751 | 1,130 | **0.65** |

> 3/8처럼 알림이 폭증한 날은 변경/알림 비율이 낮음 → **진짜 풀부킹이 지배적**이었던 날. 반면 일반일에는 변경이 알림보다 많아 "가짜 풀부킹" 비중이 높을 것으로 추정.

| 지표 | 수치 |

> - 현재 일평균 UNLIMIT JP 변경: ~1,299건 (109,083 / 84일)
> - 캡 적용 후 예상: ~529건
> - **예상 감소: ~770건/일 (59.3% 감소)**
>
> PRD 목표 (1,450→700건) 대비 **더 큰 감소 효과**가 예상됨. 다만 전체 JP 변경(non-UNLIMIT 포함)은 약 1,450건이므로, non-UNLIMIT ~150건은 유지 → 최종 예상 일 변경: ~679건.
