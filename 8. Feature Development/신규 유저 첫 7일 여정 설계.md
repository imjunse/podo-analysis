# 신규 유저 첫 7일 여정 설계

> **목적**: 결제 완료부터 7일차까지 유저 경험을 하나의 플로우로 정의하고, 3개 기능의 스코프와 핸드오프를 명확히 한다.
> **관련 PRD**: 결제→예약 원탭 연결 (Squad A) / 신규 유저 온보딩 퍼널 (Squad C) / 첫 7일 넛지 시퀀스 (Squad C)
> **참여 PO**: 박선영 (Squad A), 이준서 (Squad C)

---

## 1. 왜 이 문서가 필요한가

현재 결제 후 유저 여정이 **설계되어 있지 않다.**

| 현재 (AS-IS) | 데이터 |
|---|---|
| 결제 성공 화면 → 끝. 다음 안내 없음 | 결제 퍼널 69% 이탈 |
| 유저가 알아서 홈으로 돌아감 | — |
| 홈에서 알아서 예약 메뉴를 찾아야 함 | 1회차 이탈 82.6% |
| 온보딩 플로우 없음 (플래그 0개) | — |
| 첫 수업까지 아무 넛지 없음 | M3 잔존 45% |

3개 기능을 각각 만들더라도 **유저 입장에서 하나의 흐름**이어야 한다. 이 문서는 그 흐름을 정의한다.

---

## 2. 목표 상태 (TO-BE) 유저 플로우

```
결제 완료 ──→ 예약 원탭 ──→ 온보딩 ──→ 홈 진입 ──→ 7일 넛지 ──→ 습관 형성
[Squad A]     [Squad A]    [Squad C]  [Squad C]   [Squad C]
T+0분         T+1분        T+3분      T+10분      Day 1~7
```

### T+0: 결제 완료 직후

| 항목 | 내용 |
|------|------|
| **화면** | 결제 성공 화면 |
| **핵심 액션** | "바로 첫 수업 예약" CTA |
| **동작** | ticketId로 `/booking` 직행 |
| **담당** | Squad A — 결제→예약 원탭 연결 |
| **이 단계의 목표** | 결제→예약 이탈을 막는다 |

### T+1분: 예약 화면

| 항목 | 내용 |
|------|------|
| **화면** | 예약 화면 (시간/튜터 선택) |
| **핵심 액션** | 첫 수업 시간 + 튜터 선택 |
| **담당** | Squad A 스코프 (기존 예약 화면 활용) |
| **이 단계의 목표** | 첫 수업을 확정한다 |

### 핸드오프 1: 예약 완료 → 온보딩 진입

> **PO 합의 필요** — 아래 두 안 중 선택

| | A안: 예약 완료 화면에서 연결 | B안: 홈 진입 후 온보딩 모달 |
|---|---|---|
| **플로우** | 예약 완료 → "서비스 시작하기" 버튼 → 온보딩 | 예약 완료 → 홈 → 온보딩 풀스크린 모달 |
| **장점** | 끊김 없는 플로우 | 홈 화면을 기준으로 통일 가능 |
| **단점** | 예약 완료 화면에 Squad C 로직 침투 | 홈 진입 후 바로 모달 = 거부감 가능 |
| **추천** | **A안 추천** — 유저가 "다음 할 일"을 찾는 시점이라 자연스러움 | |

### T+3분: 온보딩 퍼널

| 항목 | 내용 |
|------|------|
| **화면** | 온보딩 전용 플로우 (풀스크린) |
| **단계** | 레벨테스트 → 학습 목표 설정 → 서비스 가치 안내 → 완료 |
| **담당** | Squad C — 신규 유저 온보딩 퍼널 |
| **이 단계의 목표** | 유저가 "이 서비스가 나한테 맞겠다"는 확신을 갖는다 |
| **스킵 정책** | 각 단계 스킵 가능하되, 레벨테스트는 강력 권장 |
| **재진입** | 중간 이탈 시 다음 앱 진입 때 이어서 진행 |

### 핸드오프 2: 온보딩 완료 → 홈 + 넛지 시작

| 항목 | 내용 |
|------|------|
| **트리거** | 온보딩 마지막 단계 완료 (또는 스킵으로 전체 종료) |
| **동작** | `onboarding_completed` 이벤트 발행 → 넛지 시퀀스 등록 |
| **홈 화면** | "첫 수업까지 D-N" 카운트다운 카드 노출 |

### Day 0~7: 넛지 시퀀스

| Day | 메시지 | 채널 | 조건 |
|:---:|--------|:----:|------|
| **0** | "첫 수업이 예약됐어요! 예습해볼까요?" | 인앱 + 푸시 | 예약 완료 상태일 때 |
| **1** | "내일 수업 준비됐나요? 예습 미리보기" | 푸시 | 수업 전날 |
| **2** | 첫 수업 완료 → "AI 리포트가 도착했어요" | 인앱 토스트 | 수업 완료 이벤트 |
| **3** | "두 번째 수업도 예약해볼까요?" | 푸시 | 2회차 미예약 시 |
| **5** | "벌써 N시간 학습했어요!" | 푸시 | 학습 기록 기반 |
| **7** | "일주일 학습 리포트" | 인앱 + 푸시 | 전체 |

**담당**: Squad C — 첫 7일 넛지 시퀀스

## 3. 기능별 스코프 정의

| 기능 | 스쿼드 | 시작점 | 끝점 | 배포 시기 |
|------|:------:|--------|------|:---------:|
| 결제→예약 원탭 연결 | A | 결제 성공 화면 | 예약 완료 화면 | **3M 전반 (퀵윈)** |
| 신규 유저 온보딩 퍼널 | C | 예약 완료 직후 | 홈 화면 진입 | 3M~4M |
| 첫 7일 넛지 시퀀스 | C | 온보딩 완료 | Day 7 | 4M |

### 독립 배포 가능 여부

- **결제→예약 원탭 연결**: 독립 배포 가능. 온보딩 없어도 단독으로 가치 있음 (결제 후 예약 이탈 감소)
- **온보딩 퍼널**: 독립 배포 가능. 결제→예약 연결 없어도 홈 진입 시점에 온보딩 트리거 가능
- **넛지 시퀀스**: 독립 배포 가능. 온보딩 없어도 결제 완료 이벤트 기반으로 동작 가능

> 각 기능이 독립 배포 가능하되, **세 기능이 모두 붙었을 때 풀 임팩트**가 나온다.

## 4. 핸드오프 상세 & PO 합의 사항

### 합의 1: 예약 완료 → 온보딩 연결 방식

- **결정 필요**: A안(예약 완료 화면에서 직접 연결) vs B안(홈 경유)
- **관련 PO**: 박선영 (예약 완료 화면 소유) + 이준서 (온보딩 소유)
- **결정 기준**: 예약 완료 화면의 현재 구조, 온보딩 진입 UX

### 합의 2: 온보딩 완료 정의

- **결정 필요**: 온보딩 "완료"의 기준
  - 모든 단계를 거쳐야 완료인가?
  - 스킵해도 완료로 치는가?
  - 중간 이탈은?
- **영향**: 넛지 시퀀스 시작 트리거에 직결

### 합의 3: 넛지 Day 2 — AI 리포트 안내 소유권

- **결정 필요**: 수업 완료 후 AI 리포트 안내를 넛지 시퀀스(Squad C)에서 보낼지, AI 리포트 도달 경로 개선(Squad B)에서 보낼지
- **관련 PO**: 이준서 (넛지) + 송하림 (AI 리포트)
- **추천**: 첫 7일 내는 넛지 시퀀스가 담당, 이후는 Squad B 도달 경로가 담당

### 합의 4: 기존 유저 처리

- **결정 필요**: 이미 결제하고 사용 중인 유저에게 온보딩을 소급 적용하는가?
- **추천**: 소급 미적용. 넛지 시퀀스의 일부(학습 리포트 등)만 선별 적용 검토.
