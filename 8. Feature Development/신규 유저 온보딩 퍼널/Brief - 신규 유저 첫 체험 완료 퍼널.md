# Brief: 신규 유저 첫 체험 완료 퍼널

> 이 문서는 PRD v1.4 기반으로, **미팅 전에 팀 전원이 읽고 오면 대부분의 궁금증이 해소**되도록 작성한 종합 브리프입니다.

---

## 1. 핵심 지표 정의 및 현황

### North Star Metric

> **체험 레슨 완료율** = 체험 레슨 정상 완료 유저 / 가입 유저

"예약률"이 아니라 "완료율"이 North Star인 이유:
- 무료체험을 신청한 유저의 **48.4%가 수업을 완료하지 않음**
- 예약만 늘려봐야 완료가 안 되면 정규 결제 전환 기반이 만들어지지 않음
- **수업을 실제로 듣게 만드는 것**이 전환 임팩트가 훨씬 큼

### 퍼널 현황 (MySQL 유저 기반 코호트, 2026-01-01 ~ 03-04)

```
가입          체험결제(신청)       체험완료
7,452명  →    4,916명         →    2,536명
         66.0%              51.6%
                     (= 가입 대비 34.0%)
```

| 지표 | 현재값 | 데이터 소스 | 목표 |
|---|---|---|---|
| **체험 레슨 완료율** (North Star) | **34.0%** | MySQL 코호트: 가입 7,452 → 체험완료 2,536 | **50% 이상** |
| 무료체험 신청률 | **66.0%** | MySQL 코호트: 가입 7,452 → 체험결제 4,916 | 75% 이상 |
| 체험 구매 클릭률 | **74.0%** | ClickHouse: 가입 1,132 → 클릭 838 (2/23~3/4) | 80% 이상 |
| 체험결제→체험완료율 | **51.6%** | MySQL 코호트: 체험결제 4,916 → 체험완료 2,536 | **70% 이상** |
| 체험→정규 전환율 | **38.0%** | ClickHouse: 레슨페이지 516 → 결제 196 (2/23~3/4) | 45% 이상 |

### 데이터 소스 정리

| 용도 | 소스 | 이유 |
|---|---|---|
| **퍼널 전환율** (source of truth) | **MySQL** (GWATOP) | GT_USER, GT_PAYMENT_INFO, GT_CLASS 테이블 — 유저 기반(코호트) 전환율 산출. 가입월 코호트 기준으로 각 유저의 전환 여부를 추적하여 월 경계를 넘는 전환도 정확히 반영 |
| **행동 패턴 분석** | **ClickHouse** (podo_events) | 시간대, 페이지 이동, 클릭 이벤트 등 행동 데이터 |

> ClickHouse의 `meet_connected` 이벤트는 2026년 3월부터만 기록되어 있어, 참석/노쇼 세부 분석은 MySQL 기반으로 산출합니다. 모든 퍼널 수치는 **유저 기반(코호트) 전환율** — 스냅샷 기반(각 테이블 날짜 컬럼별 집계)과 달리 실제 전환율을 정확히 반영합니다.

### 일평균 신규 가입

- **~118명/일** (MySQL 실측: 7,452명 ÷ 63일)
- A/B 테스트 시 각 그룹 200명 도달: **~3.4일**, 1,000명: **~17일**

## 2. 지표에 영향을 주는 요소

### 최대 병목: 체험결제 → 체험완료 (48.4% 이탈)

이 구간에서 절반 가까이 이탈합니다. 원인은 크게 두 가지입니다.

#### 원인 1: CTA 버그 (주요 원인, 현재 수정 중)

- 2025년 하반기 React 전환 과정에서 **"수업 입장하기" CTA가 정상 작동하지 않는 버그** 발생
- ClickHouse 기준 2월에는 `meet_connected` 이벤트가 **0건** — 트래킹 추가 시점 또는 CTA 버그와 연관
- **이 버그 하나가 노쇼의 가장 큰 원인으로 추정됨**
- 현재 수정 중이며, 수정 후 실측 데이터를 먼저 확보

#### 원인 2: UX 구조적 문제

| UX 문제 | 설명 |
|---|---|
| 예약~수업 사이 빈 시간 | 예약 후 수업까지 유저를 붙잡아두는 장치 없음 |
| 1회 체험 기회 소멸 | 노쇼 시 체험 기회가 사라짐 — 재시도 안내 부재 |
| 레슨방 입장 불안 | 첫 수업 전 긴장감을 완화하는 UX 없음 |
| 서비스 가치 미전달 | 가입 → 바로 예약/결제 유도. "이 서비스가 왜 필요한지" 알려주지 않음 |
| 레벨 매칭 부정확 | 자가 선택 5단계 — 진단 없이 유저가 직접 고름 |
| 이탈 후 재진입 경로 부재 | 예약 페이지 이탈 시 다시 유도하는 장치 없음 |

### 가입 → 체험결제 구간 (34.0% 이탈)

코호트 분석으로 가입자의 **34.0%(2,536명)가 무료체험조차 신청하지 않는** 이탈이 새롭게 확인되었습니다 (스냅샷 기반에서는 22.5%로 과소평가됨). 원인:
- 로그인 즉시 `/subscribes/trial`로 자동 리다이렉트 — 서비스 설명 없이 결제부터 요구
- `trial_free` 플래그가 ON이면 0원 결제(무료)인데도, 결제 UX가 심리적 허들로 작용

### 행동 패턴 데이터 (ClickHouse, 2/23~3/4)

| 패턴 | 데이터 | 시사점 |
|---|---|---|
| 예약 피크 시간대 | 22시(1,101건) > 23시(1,092건) > 21시(961건) > 19시(858건) | 저녁~심야에 집중 |
| 예약-수업 시간 갭 | 1~6시간이 최다(31.3%), 86.3%가 24시간 내 | 당일/익일 수업 선호 |
| 결제 방식 선호 | 일시불 49.0% vs 구독 19.8% (일시불 2.5배 선호) | 결제 UI에서 일시불 우선 |
| 예약 요일 | 화·수 예약이 가장 많음 | — |

## 3. 서비스 플로우 전반 딥다이브 및 진단

### 전체 퍼널 구조

```
가입 → [온보딩] → 체험 예약 → [대기·리마인드] → 레슨방 입장 → [체험 수업] → 체험 완료 → [결제 유도]
       ──────────  ─────────────────────────────  ──────────────────────  ──────────────
       Sub-PRD 1        Sub-PRD 2                       Sub-PRD 3              Sub-PRD 4
       (이번 PRD)    (예약→체험 완료)                  (레슨 경험)            (결제 전환)
```

### 3-1. AS-IS: 현재 플로우 진단

**현재 흐름:**
1. 카카오/Apple 로그인
2. `isTrialPopUpOpenCondition` 체크 → `redirection=trial` 세팅
3. `/subscribes/trial` 자동 리다이렉트
4. 언어 선택 → 레벨 자가 선택 → 결제 → 예약

**문제 진단:**

| 단계 | 문제 | 심각도 |
|---|---|---|
| 로그인 직후 | 서비스가 뭔지도 모르는 상태에서 바로 결제/예약 페이지로 이동 | 높음 |
| 언어 선택 | 단순 카드 2개 (EN/JP) — 문제없음 | 낮음 |
| 레벨 선택 | 5단계 자가 선택 — 유저가 자기 레벨을 정확히 모름 | 중간 |
| 결제 | `trial_free` ON이면 0원이지만, 결제 UI 자체가 심리적 허들 | 중간 |
| 예약 후 | 대기 시간 동안 할 게 없음. 리마인드만 있음 | 높음 |
| 수업 입장 | CTA 버그로 입장 불가 (수정 중) | **치명적** |
| 수업 완료 후 | 결제 유도가 카카오톡 알림톡 — 앱으로 다시 안 돌아옴 | 중간 |

### 3-2. TO-BE: 새 온보딩 플로우 설계

```
AS-IS (/subscribes/trial)              TO-BE (온보딩 플로우)
─────────────────────────              ────────────────────────
(없음 — 바로 예약 페이지)        →      [Intro] 앱 소개 2~3화면           ★ 신규
언어 선택 (EN/JP 카드)          →      [Step 1] 환영 + 학습 언어 선택
                                       [Step 2] 학습 목표 설정           ★ 신규
레벨 자가 선택 (5단계)          →      [Step 3] 시작 레벨 추천 (5문항)  ★ 개선
                                       [Step 4] 맞춤 결과 + 가치 안내     ★ 신규
결제 + 예약 CTA                →      [Step 5] 무료 체험 레슨 예약       ★ 간소화
                                              + 예약 완료 후 CTA 슬롯    ★ 신규
```

**전체 소요 시간: ~2분 15초**

| Step | 무엇을 하는가 | 왜 필요한가 | 소요 시간 |
|---|---|---|---|
| Intro | 앱 소개 2~3화면 스와이프 | 브랜드 첫인상 + 서비스 핵심 가치 전달 | ~15초 |
| Step 1 | 환영 + 학습 언어 선택 | 개인화 시작. 카드 탭 시 자동 진행 | ~15초 |
| Step 2 | 학습 목표 설정 (복수선택, 최대 2개) | 유저 데이터 수집 + 동기부여 | ~20초 |
| Step 3 | 시작 레벨 추천 5문항 | 기존 퀴즈 인프라 재활용. **잠금이 아닌 추천** — 모든 레벨 자유 선택 가능 | ~60초 |
| Step 4 | 추천 결과 + 가치 안내 | ① 추천 레벨 ② 자유 선택 안내 ③ "첫 수업 후 AI가 정밀 분석" 예고 → 체험 동기 극대화 | ~20초 |
| Step 5 | 무료 체험 레슨 예약 | 기존 체험레슨 예약 화면 재활용 (7일 가로 스크롤 + 시간 그리드 + 확인 바텀시트) | ~15초 |

**핵심 설계 원칙:**

1. **레벨 추천은 잠금이 아니라 출발점** — 정밀 레벨링은 첫 수업 후 AI 리포트(FLUENCY/VOCAB/GRAMMAR/COMPLEXITY)에 위임
2. **Step 1~4는 건너뛰기 없음** — ~2분의 라이트한 플로우이므로 스킵 불필요
3. **Step 5만 "나중에 할게요" 제공** — 건너뛴 유저는 홈 배너로 재유도
4. **중간 이탈 시 이어서 진행** — localStorage 기반 상태 저장 (동일 기기)

### 3-3. Sub-PRD 2: 예약 → 체험 완료

> CTA 버그 수정이 **가장 크게 체험 완료율을 움직일 카드**. 버그 수정 후 실측 데이터 확보 → 추가 개선 범위 조정.

**상태별 홈 화면 설계 (4가지 상태):**

| 상태 | 화면 구성 |
|---|---|
| **대기 중** | 레슨 상태 카드(일시/커리큘럼) + AI학습·교재 미리보기 CTA + 프로그레스바(2/4) |
| **레슨 10분 전** | "레슨방이 열렸어요!" + "무료체험 입장하기" CTA + 프로그레스바(3/4) |
| **튜터 노쇼** | 사과 안내 + 체험권 1회 복구 + 휴먼 케어콜 |
| **레슨 완료** | **인앱 진단 리포트** 안내 + "내 진단 리포트 보기" CTA |

**주요 변경점:**
- 진단 리포트를 카카오톡 알림톡 → **인앱 리포트**로 변경 (앱 이탈 방지)
- 예약 완료 직후 교재·AI학습 CTA 제공 (대기 시간 경험)

### 3-4. Sub-PRD 3: 체험 완료 → 결제 유도

**v1 MVP:** 인앱 진단 리포트 하단에 결제 CTA 1개 추가

- "지금 시작하면 첫 수강 혜택을 받을 수 있어요" → 수강권 구매 페이지 이동
- 홈 화면 전체 재구성은 v2로 미룸

> 체험 완료 후 유저가 리포트를 확인하는 시점이 **결제 전환 동기가 가장 높은 순간**이므로, 여기에 집중.

### 3-5. 기술 아키텍처 요약

| 영역 | 핵심 내용 |
|---|---|
| 피처 플래그 | `onboarding_v1` (단일 플래그로 전체 ON/OFF). 기존 `trial_free` 플래그와 독립적 |
| 프론트엔드 | 기존 파일 5개 수정 + `/onboarding` 라우트 신규 생성. Step 5 예약 UI는 기존 화면 재활용 |
| 백엔드 | 신규 API 2개: 레벨 추천 결과 저장, 학습 목표 저장 |
| 온보딩 상태 | localStorage 기반 (서버 저장 아님). ~1분 플로우이므로 서버 부하 불필요 |
| 데이터 모델 | `onboarding_result` 테이블 — 완료 결과만 저장 (진행 상태는 클라이언트) |

## 4. 트렌드/벤치마크 리서치

### 업계 벤치마크 비교

| 지표 | 업계 벤치마크 | PODO 현재 | 평가 |
|---|---|---|---|
| **무료 체험 신청률** | B2C 일반 30~50%, 교육 앱 우수 사례 80%+ | **66.0%** | B2C 평균 이상이나, 교육 앱 우수 사례 대비 **개선 여지 큼** |
| **체험 완료율** (free trial activation) | B2C 40~60%가 양호 | **34.0%** (= 51.6% activation) | 보통 이하 (CTA 버그 감안 시 잠재력 높음) |
| **체험→정규 전환율** | 별도 벤치마크 조사 필요 | **38.0%** | Sub 3 스코프 (이 Brief 범위 밖) |
| Day 30 리텐션 (교육 앱) | **~2%** (교육 앱 평균) | 별도 측정 필요 | — |

**시사점:**
- PODO의 **무료체험 신청률(66.0%)은 B2C 평균(30~50%) 대비 높지만, 교육 앱 우수 사례(80%+) 대비 개선 여지가 큼** — 가입자 3명 중 1명이 체험조차 신청하지 않는 상태
- 문제는 **신청 후 완료까지의 전환**. 여기서 48.4%가 이탈하는데, CTA 버그가 주요 원인이므로 버그 수정만으로 유의미한 개선 가능
- **체험→정규 전환율 38%** — 체험을 완료한 유저의 결제 전환은 별도 스코프(Sub 3)이나, "수업을 들은 유저는 결제한다"는 점에서 **체험 완료율 개선의 ROI가 높음**

### 듀오링고 온보딩 사례

듀오링고는 온보딩의 교과서적 사례입니다:

| 듀오링고 전략 | PODO 적용 |
|---|---|
| **가입 전에 먼저 경험** (gradual engagement) — 회원가입 전 번역 문제 풀기 | 온보딩 5문항으로 "레벨 추천"을 받아보는 경험 제공 |
| **개인화된 첫 경험** | Step 1~4에서 언어/목표/레벨을 수집하여 맞춤 결과 제시 |
| **$0 표현** — "Try for $0"이 "Free trial"보다 전환율 높음 | `trial_free` ON 시 "무료 체험" 강조 |
| **분기별 300건 이상의 실험** | A/B 2그룹 테스트 + 단계별 이탈 모니터링 |
| **MAU→Premium 3% → 8.8%** (5년간 176% 성장) | 체험 완료율 34.0% → 50% 목표 (+47% 성장) |

## 5. 개선 방향 및 우선순위

### 임팩트 순서로 본 우선순위

```
                        높은 임팩트
                            │
    ┌───────────────────────┼───────────────────────┐
    │                       │                       │
   1순위               2순위               3순위
  CTA 버그 수정       온보딩 플로우        결제 유도
  (Sub 2)            (Sub 1)             (Sub 3)
  체험완료율          무료체험신청률        정규전환율
  직접 영향           75% 목표            45% 목표
                     체험완료율 간접
```

#### 1순위: CTA 버그 수정 (현재 진행 중)

- **왜 1순위인가:** 체험결제→체험완료 48.4% 이탈의 **최대 원인**. 온보딩을 아무리 잘 만들어도 수업 입장이 안 되면 무의미
- **기대 효과:** 버그 수정만으로 체험 완료율이 유의미하게 상승할 가능성 높음
- **액션:** 수정 중. 수정 후 실측 데이터 확보 → Sub 2 추가 개선 범위 재조정

#### 2순위: 온보딩 플로우 신규 구축 (Sub 1, W1~W8)

- **왜 2순위인가:** 예약률(66.0% → 75%)과 체험 완료 동기를 동시에 끌어올리는 구조적 개선
- **핵심 변화:**
  - 서비스 가치 전달 (Intro)
  - 개인화된 레벨 추천 (Step 3~4)
  - 체험 예약 UX 간소화 (Step 5)
  - "AI 진단 리포트" 예고 → 수업을 들어야 할 명확한 동기
- **개발 기간:** W1~W8 (8주)
- **A/B 테스트:** Control vs Treatment, 최소 2주, ~118명/일

#### 3순위: 결제 유도 (Sub 3, W8~W10)

- **왜 3순위인가:** 이미 체험→정규 전환율이 38%로 양호. 인앱 리포트 내 CTA 추가(v1 MVP)로 점진적 개선
- **v1 스코프:** 리포트 하단 CTA 1개 — "지금 시작하면 첫 수강 혜택을 받을 수 있어요"
- **v2 검토:** 홈 화면 전체 재구성 (혜택 카드, 프로모션 배너)

## 부록: FAQ — 자주 나올 수 있는 질문

**Q. 왜 예약률이 아니라 완료율이 North Star인가요?**
→ 이미 66.0%가 예약을 하고 있는데, 그중 절반 가까이가 수업을 안 듣습니다. 예약만 늘려봐야 완료가 안 되면 정규 결제로 이어지지 않습니다.

**Q. 온보딩 2분이 추가되면 이탈이 더 늘지 않나요?**
→ 현재는 서비스가 뭔지 모른 채 결제/예약 페이지로 이동합니다. 2분 투자로 "왜 이 서비스가 필요한지"를 이해시키면 예약 품질(실제 수업 참여)이 올라갈 것으로 기대합니다. 이탈 증가 가능성은 A/B 테스트로 검증합니다.

**Q. 레벨 추천이 틀리면 어떡하나요?**
→ 설계 자체가 "잠금"이 아닌 "출발점 추천"입니다. 유저는 어떤 레벨/주제든 자유롭게 선택 가능하고, 정밀 레벨링은 첫 수업 후 AI 리포트가 담당합니다.

**Q. `trial_free` 플래그랑 `onboarding_v1` 플래그 관계는?**
→ 독립적입니다. `onboarding_v1`은 온보딩 플로우 전체 ON/OFF, `trial_free`는 결제 무료 여부. 조합에 따라 4가지 케이스가 존재합니다:
- `onboarding_v1=ON` + `trial_free=ON` → 온보딩 + 무료 예약
- `onboarding_v1=ON` + `trial_free=OFF` → 온보딩 + 유료 결제 포함 예약
- `onboarding_v1=OFF` + `trial_free=ON` → 기존 플로우 + 무료
- `onboarding_v1=OFF` + `trial_free=OFF` → 기존 플로우 + 유료

**Q. 왜 3그룹이 아니라 2그룹 A/B인가요?**
→ 유저가 이미 모든 레벨/주제를 자유 선택할 수 있고, 첫 수업 후 AI 리포트로 정밀 레벨링이 되므로, "레벨 추천 생략(자가선택 유지)" 그룹을 별도로 둘 실익이 없습니다.

**Q. Sub 2에서 무료 체험 횟수를 3회로 늘리는 건 어떻게 됐나요?**
→ 기각되었습니다. 노쇼의 주원인이 CTA 버그로 확인되어, "한 번 밖에 기회가 없어서 부담"이라는 전제가 소멸했습니다. 버그 수정 후에도 노쇼율이 높으면 재논의합니다.
