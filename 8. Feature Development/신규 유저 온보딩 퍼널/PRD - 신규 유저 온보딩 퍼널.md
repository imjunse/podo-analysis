# PRD: 신규 유저 첫 체험 완료 퍼널 (마스터)

## 1. 개요

### 한 줄 요약
신규 유저의 **가입 → 온보딩 → 무료 체험 예약 → 체험 레슨 완료**까지의 전체 퍼널을 재설계하여, 체험 레슨 완료율을 끌어올리고 정규 결제 전환 기반을 만든다.

### 전체 퍼널 정의

```
스플래시 → 로그인 → 가입 → [온보딩] → 체험 구매 → 예약 → [대기·리마인드] → 레슨방 입장 → [체험 수업] → [리뷰·NPS] → 체험 완료 → [결제 유도]
                           ──────────  ────────────────────────────────────────────────────────────────────────  ──────────────
                           Sub-PRD 1                                    Sub-PRD 2                                     Sub-PRD 3
                           (온보딩 플로우)                          (구매→예약→체험 완료)                                (결제 전환)
```

| Sub-PRD | 스코프 | 상태 |
|---|---|---|
| **Sub 1: 온보딩 플로우** | 가입 → 온보딩 → 무료 체험 예약 | 설계 완료 (v0.5), 이 문서 §5 |
| **Sub 2: 예약 → 체험 완료** | 대기 경험, 리마인드, 레슨방 입장, 노쇼/취소 처리, 인앱 진단 리포트 | 초안 (이 문서 §12) |
| **Sub 3: 체험 완료 → 결제 유도** | 체험 완료 후 홈 화면, 정규레슨 결제 전환 | 초안 (이 문서 §13) |

### 배경

현재 신규 유저의 첫 경험은 **체험 레슨 구매 페이지**(`/subscribes/trial`)로의 자동 리다이렉트다. 이 페이지에서 언어 선택 → 레벨 자가 선택 → 결제(또는 `trial_free` 플래그 ON 시 0원 결제)를 처리한다. **구매와 예약은 별도 단계**로, 구매 완료 후 홈(튜토리얼 가이드) → 체험 레슨 목록(`/lessons/trial`) → 예약(`/booking`)으로 이어지는 다단계 플로우다.

문제는 이 플로우가 **서비스가 무엇인지, 왜 나한테 필요한지** 전달하지 않고 바로 결제를 요구한다는 점이다. 사실상 이 구매 페이지가 유일한 "온보딩"이지만, 개인화도 가치 전달도 없다.

MySQL 유저 기반(코호트) 실측 기준(2026-01-01~03-04), 가입 **7,452명** 중 무료체험을 신청(결제)한 유저는 **4,916명(66.0%)**이고, 그중 체험 레슨을 실제 완료한 유저는 **2,536명**이다. **가입 → 체험 완료 전환율은 34.0%**로, 무료체험을 신청한 유저의 **48.4%가 체험 수업을 완료하지 않는 것이 최대 이탈 구간**이다. 또한 가입자의 **34.0%(7,452 − 4,916 = 2,536명)가 무료체험조차 신청하지 않는** 이탈도 코호트 분석으로 새롭게 드러났다 (스냅샷 기반에서는 22.5%로 과소평가됨). *(참고: 체험 완료 유저 수와 체험 미신청 유저 수가 우연히 동일한 2,536명이다.)*

이 이탈에는 두 가지 원인이 복합적으로 작용한다:

1. **기술 버그 (주요 원인)**: 2025년 하반기 React 전환 과정에서 "수업 입장하기" CTA가 정상 작동하지 않는 버그가 발생했다. ClickHouse 기준 2월에는 `meet_connected` 이벤트가 **0건**이며 3월부터만 집계되어, 트래킹 추가 시점 또는 CTA 버그와 연관된 것으로 추정된다. **이 버그는 현재 수정 중이다.**
2. **UX 문제**: 예약과 수업 사이 시간에 유저를 붙잡아두는 장치가 없고, 무료 체험이 1회뿐이라 노쇼 시 기회가 소멸되며, 레슨방 입장 경험도 불안 해소가 되지 않는다.

> ⚠️ **데이터 참고**: 모든 퍼널 수치는 **유저 기반(코호트) 전환율** — 가입월 코호트 기준으로 각 유저의 전환 여부를 추적한 수치다. 스냅샷 기반(각 테이블 날짜 컬럼별 집계)과 달리 월 경계를 넘는 전환도 가입월에 귀속시켜 실제 전환율을 정확히 반영한다. MySQL이 퍼널 전환율의 source of truth이며, ClickHouse는 행동 패턴 분석(시간대, 페이지 이동 등)에만 활용한다.

**온보딩을 아무리 잘 만들어도 예약 이후 퍼널이 무너지면 전체 임팩트가 제한적**이다. 단, CTA 버그 수정만으로도 체험 미완료율이 유의미하게 개선될 수 있으므로, 버그 수정 후 실측 데이터를 먼저 확보한 뒤 Sub 2의 추가 개선 범위를 조정한다.

### 대상 유저
> **체험 전 신규 유저** — 앱을 처음 다운로드하고 가입한 유저. 체험 레슨도 아직 하지 않은 상태.

이 PRD는 체험 전 신규 유저만 다룬다. 정규 결제 유저의 결제→예약 플로우(Squad A 담당)와는 별도 경로다.

**핵심 기술 요소:**
- `trial_free` 피처 플래그: 이미 구현 완료. ON 시 결제 없이 0원 체험 가능
- `isFirstTrialSubscribe`: `selectedSubscribe.subPrice === 500`으로 판별
- 피처 플래그 시스템: `@podo/flags/react`의 `useFlagsCompat` 사용
- ⚠️ **`TRIAL_FREE` 수업방 미생성**: `PaymentType=TRIAL`은 결제 시 `GT_CLASS`(수업방)를 자동 생성하지만, `TRIAL_FREE`는 수업방을 자동 생성하지 않음. 현재는 예약(`/booking`) 시점에 별도 처리. TO-BE 온보딩에서 예약 플로우 설계 시 이 차이를 고려해야 함

## 2. 목표 & 성공 지표

### 목표
> 신규 유저의 **가입 → 체험 레슨 완료**까지의 전체 퍼널을 재설계하여, 체험 완료율을 끌어올리고 서비스 가치를 체험한 유저 모수를 늘린다.

### North Star Metric

> **체험 레슨 완료율** = 체험 레슨 정상 완료 유저 / 가입 유저

예약률만 올려서는 안 된다. 무료체험을 신청한 유저의 **48.4%가 체험 수업을 완료하지 않는** 현재 구조에서는, 예약률 개선보다 **예약한 유저가 실제로 수업을 듣게 만드는 것**이 전환 임팩트가 더 크다.

### Non-Goal (이번에 하지 않는 것)
- 정규 구독 온보딩 (체험 → 정규 전환 플로우의 결제 UI 자체는 Squad A 담당)
- 기존 유저 대상 온보딩 소급 적용
- 튜터 앱 측 온보딩 변경
- 앱푸시/넛지 시퀀스 (별도 과제 "첫 7일 넛지 시퀀스"로 분리)
- 튜터 매칭 알고리즘 변경 (레슨 품질은 이 PRD 스코프 밖)

> **v0.5 대비 변경**: "정규 구독 결제 유도"와 "체험 예약 후 대기 시간 경험"이 Non-Goal에서 제거됨. 체험 완료율을 올리려면 예약 이후 경험(Sub 2)과 체험 완료 후 결제 유도(Sub 3)가 이 퍼널 안에 있어야 하기 때문.

## 3. 유저 스토리

### 대상 유저 정의

```
이 PRD의 대상: 체험 전 신규 유저
  → 앱 첫 다운로드 → 카카오/Apple 로그인 → [온보딩] → 무료 체험 예약

별도 PRD (Squad A):  정규 결제 유저
  → 결제 완료 → 예약 원탭 연결 → 홈 진입
```

### 핵심 시나리오

**US-1: 신규 유저의 첫 앱 실행**
> 앱을 처음 연 유저로서, 로그인하기 전에 이 서비스가 뭔지 알고 싶고, 나에게 맞는지 직접 경험해보고 싶다. 충분히 투자한 뒤에 가입하면 더 확신이 든다.

수용 기준:
- 앱 첫 실행 시 로그인 없이 온보딩 플로우가 시작된다 (Delayed Sign-up)
- 앱 소개 → 학습 언어, 목표, 레벨을 설정할 수 있다
- 설정 결과에 따른 개인화 안내를 받는다
- 추천 결과 확인 후 로그인(가입)하여 **무료 체험 레슨을 예약**할 수 있다

**US-2: 온보딩 중간 이탈 후 재진입**
> 온보딩 중 앱을 닫은 유저로서, 다시 앱을 열면 이어서 진행하고 싶다. 처음부터 다시 하기 싫다.

수용 기준:
- 진행 상태가 localStorage에 저장된다
- 재진입 시 마지막 완료 단계 다음부터 재개된다 (동일 기기 한정)

**US-3: 2회차 이후 로그인**
> 온보딩을 완료한 유저로서, 다시 로그인하면 바로 홈 화면에 들어가고 싶다.

수용 기준:
- `onboarding_completed = true`이면 온보딩을 건너뛴다
- 홈 화면으로 직행한다

**US-4: 체험 예약 건너뛴 유저**
> 온보딩은 했지만 체험 레슨을 예약하지 않은 유저로서, 홈에서 쉽게 예약할 수 있었으면 좋겠다.

수용 기준:
- 홈 화면 상단에 "무료 체험 레슨 예약하기" 배너가 노출된다 (기존 `TrialBanner` 컴포넌트 활용)
- 배너 클릭 시 Step 5(체험 예약 화면)로 이동한다

**US-5: 체험 예약 후 수업 대기 중인 유저** *(Sub 2)*
> 체험 레슨을 예약한 유저로서, 수업 전까지 이 앱이 어떤 건지 미리 경험하고 싶고, 수업 시간을 잊지 않고 싶다.

수용 기준:
- 홈 화면에 예약 상태 카드가 노출된다 (레슨 일시, 남은 시간)
- AI 학습이나 교재를 미리 볼 수 있는 CTA가 제공된다
- 레슨 시작 전 리마인드 알림이 발송된다 (앱 푸시 알림. 앱푸시 넛지 시퀀스는 별도 과제이나, 체험 레슨 리마인드는 단일 알림으로 이 PRD 스코프 내에서 처리)

**US-6: 체험 완료 후 진단 리포트를 확인하려는 유저** *(Sub 2)*
> 체험 레슨을 마친 유저로서, AI가 분석한 내 실력 진단 리포트를 앱에서 바로 확인하고 싶다.

수용 기준:
- 체험 완료 후 인앱 진단 리포트가 생성된다
- 홈 화면에서 진단 리포트로 바로 이동할 수 있는 CTA가 있다
- 리포트에 FLUENCY/VOCAB/GRAMMAR/COMPLEXITY 분석 결과가 포함된다

**US-7: 체험을 완료하고 정규 결제를 고려하는 유저** *(Sub 3)*
> 무료 체험을 마친 유저로서, 정규 수강의 혜택을 알고 결제할지 판단하고 싶다.

수용 기준:
- 인앱 진단 리포트 하단에 정규 수강 유도 CTA가 있다
- CTA 클릭 시 결제 페이지로 바로 이동할 수 있다

## 5. 온보딩 플로우 상세

> 상세 화면 설계, 단계별 구성, 트래킹 이벤트 정의는 [[온보딩 플로우 설계 초안]] 참조.

### AS-IS → TO-BE 매핑

```
AS-IS                                  TO-BE (온보딩 플로우)
─────────────────────────              ────────────────────────
스플래시 (/)                    →      스플래시 (유지)
(없음 — 바로 로그인)             →      [Intro] 앱 소개 2~3화면           ★ 로그인 전 신규
로그인 (/login, 카카오/Apple)    →      (로그인을 Step 4 뒤로 이동)       ★ Delayed Sign-up
OAuth 콜백 → /home?redirection  →      (Intro 직후 온보딩 진입, 비로그인)  ★ 변경
언어 선택 (EN/JP 카드)          →      [Step 1] 환영 + 학습 언어 선택     (비로그인)
                                       [Step 2] 학습 목표 설정           ★ 신규 (비로그인)
레벨 자가 선택 (4단계)          →      [Step 3] 시작 레벨 추천 (6문항)  ★ 개선 (비로그인)
                                       [Step 4] 맞춤 결과 + 가치 안내     ★ 신규 (비로그인)
                                       로그인 (카카오/Apple)              ★ 여기서 가입
결제(구매) CTA                  →      [Step 5] 무료 체험 레슨 예약       ★ 간소화 (구매+예약 통합)
                                       알림 권한 요청                     ★ 신규
                                       예약 완료 화면 + CTA 슬롯          ★ 신규
```

### 플로우 요약

```
스플래시 (/)
  → [Intro] 앱 소개 — 2~3화면 스와이프           (10~15초)  ※ 비로그인
  → [Step 1] 환영 + 학습 언어 선택              (15초)     ※ 비로그인
  → [Step 2] 학습 목표 설정                      (20초)     ※ 비로그인
  → [Step 3] 시작 레벨 추천 — 6문항            (60~70초)  ※ 비로그인
  → [Step 4] 추천 결과 + 서비스 가치 안내         (20초)     ※ 비로그인
  → 로그인 (/login, 카카오/Apple)                (15초)     ★ Delayed Sign-up
  → OAuth 콜백 (서버: 온보딩 데이터 저장 + 체험 대상 판별)
  → [Step 5] 무료 체험 레슨 예약                 (15초)
  → [알림 권한] "수업 리마인드 받을까요?"          (5초)      ★ 신규
  → 예약 완료 화면 (+ 후속 경험 CTA 슬롯)
  → 홈 진입
```

> **Delayed Sign-up 설계 의도**: Step 1~4에서 ~2분 투자한 유저는 Sunk Cost Effect로 로그인 이탈률이 낮다. 듀오링고, Calm 등 주요 앱이 동일 패턴을 사용. Step 1~4 데이터는 localStorage에 임시 저장하고, 로그인 성공 후 서버로 일괄 전송한다.

| **알림 권한**        | 앱 푸시 알림 권한 획득 — 리마인드·넛지 발송 기반                                                                   | 예약 완료 직후 "수업 리마인드 알림을 받을까요?" 1회 요청. 거부해도 예약 완료 진행에 영향 없음. 향후 인앱 알림 적극 활용 기반                                          | **신규** — 현재 없음  |

### Step 3: 시작 레벨 추천 — 설계 원칙

> **레벨 추천은 잠금이 아니라 출발점이다.**

| 원칙 | 설명 |
|---|---|
| **추천, 제한 아님** | 진단 결과는 "추천 시작 레벨"이며, 유저는 이후 어떤 레벨/주제든 자유롭게 수업 선택 가능 |
| **기존 인프라 재활용** | `/api/v1/diagnosis/getQuestions`, `checkAnswer` API의 문제 타입 재활용 |
| **정밀 레벨링은 첫 수업 후** | 온보딩 진단은 대략적 추천. 정확한 레벨 분석은 첫 체험 수업 후 AI 리포트(FLUENCY/VOCAB/GRAMMAR/COMPLEXITY)에 위임 |
| **문항 수** | 6문항 확정. 난이도 분포: 초급 1 / 중급 2 / 중고급 2 / 고급 1. 60~70초 내 완료 가능 |
| **문항 타입** | 전체 4지선다 객관식. 빈칸 채우기·듣기 등 복잡한 인터랙션 배제 — 터치만으로 진행 |
| **문항 풀** | v1에서 고정 (영어 6문항, 일본어 6문항). 교육팀이 확정한 문항을 변경 없이 사용. 문항 교체·랜덤 추출은 v2 검토 |
| **UX 규칙** | 시간제한 없음 · 정답/오답 피드백 없음 (테스트가 아닌 "추천" 느낌) · 이전 문항 돌아가기 불가 · 프로그레스 바 표시 (1/6 ~ 6/6) |

```
온보딩 레벨 추천 (Step 3)  →  "시작 레벨 추천" (기본값 세팅)
                                ↓
실제 수업 선택              →  유저가 원하는 주제/난이도 자유 선택 가능
                                ↓
첫 수업 후 AI 리포트        →  FLUENCY/VOCAB/GRAMMAR/COMPLEXITY 기반 정밀 레벨 보정
```

### Step 4: 추천 결과 + 가치 안내 — 메시지 구성

Step 4에서 전달할 6가지 요소 + 로그인 CTA:

| # | 요소 | 목적 |
|---|---|---|
| ① | **레벨 바 비주얼** — 4단계(초급/중급/중고급/고급) 중 추천 위치 하이라이트 | 한눈에 "나는 여기" 인지. 진단 결과의 신뢰감 부여 |
| ② | "**[레벨]**로 시작해보세요" | 개인화된 느낌 (비로그인이므로 이름 대신 레벨만 강조) |
| ③ | **레벨 설명 1줄** — "일상적인 대화를 이해하고 간단히 표현할 수 있는 단계예요" (레벨별 상이) | 추천 레벨이 무엇을 의미하는지 즉시 파악 |
| ④ | "물론, 다른 레벨도 자유롭게 들을 수 있어요" | 선택 폭 제한 불안 해소 |

| ⑥ | "첫 수업 후 AI가 실력을 분석해서 나만의 진단 리포트를 만들어줘요" | 체험 수업을 들어야 할 동기 + 인앱 AI 진단 리포트 가치 예고 |
| **CTA** | **"무료로 시작하기"** → 로그인 화면 이동 | Delayed Sign-up: 4단계 투자 후 가입 유도. 로그인 성공 시 Step 5로 자동 진행 |

> **추천 수업 vs 홈 커리큘럼**: Step 4에서는 1개만 보여 CTA 집중. 홈 진입 후 다른 커리큘럼/레벨도 자유 탐색 가능.

### 확장 온보딩 (v1.1 — 1차 A/B 통과 후 2차 실험)

> **상태**: 설계 예정. 1차 A/B에서 코어 온보딩(Step 1~4)의 효과가 검증된 후, 2차 A/B로 추가 질문의 효과를 실험한다.

**추가 단계 (Step 2 뒤에 삽입):**

| Step | 목적 | 핵심 판단 | 소요 시간 |
|---|---|---|---|
| 2-1. 학습 시간대 선호 | 리마인드 알림 최적 시간 확보 + 개인화 데이터 | "주로 언제 공부해요?" — 아침/점심/저녁/밤 4택. Step 5 예약 시 선호 시간대를 상단 노출하는 데 활용 가능 | ~10초 |

**2차 A/B 설계:**

| 그룹 | 경험 | 온보딩 길이 |
|---|---|---|
| Control | 코어 온보딩 (v1 — Step 1~4 + 예약) | ~2분 15초 |
| Treatment | 확장 온보딩 (코어 + 시간대/빈도) | ~2분 35초 |

**의사결정 기준:**
- 확장 온보딩의 무료체험 신청률이 코어 대비 **동등 이상** (이탈 증가 없음 확인) → 롤아웃
- 이탈률 증가 → 확장 질문 제거, 코어만 유지

**데이터 활용 계획:**

| 수집 데이터 | 즉시 활용 (v1.1) | 향후 활용 (v2+) |
|---|---|---|
| 학습 시간대 | Step 5 예약 시 선호 시간대 상단 노출, 리마인드 알림 발송 시간 최적화 | 수업 추천, 넛지 시퀀스 타이밍 |
| 학습 빈도 | 체험 완료 후 수강권 추천에 참고 (Sub 3) | 학습 계획 자동 생성, 수강권 타입 추천 |

**데이터 모델 확장 (v1.1):**

```
onboarding_result {
  ...기존 필드...
  preferred_time_slot  -- morning / afternoon / evening / night (nullable)
  preferred_frequency  -- daily / 3_4_per_week / 1_2_per_week / undecided (nullable)

> **왜 순차 실험인가?** 일평균 가입 ~118명으로 3그룹 동시 실험 시 각 그룹 충분한 표본 확보에 25일+ 소요. 코어 온보딩 효과를 먼저 확인한 뒤 확장하는 것이 리스크가 낮다.

### Step 5: 무료 체험 예약 — 설계 변경

> **v1.2 대비 변경**: "추천 시간 4개 우선 노출" 패턴 제거 → **일주일간 예약 가능한 모든 시간대 노출**로 변경. 기존 PODO 체험레슨 예약 화면과 유사한 형태.

**예약 UI 구성:**

| 순서 | 요소 | 설명 |
|---|---|---|
| 1 | 무료 체험 안내 | "무료 체험 레슨을 예약해보세요" |
| 2 | 날짜 선택 (7일) | 오늘 포함 7일간의 날짜를 가로 스크롤로 노출. 오늘이면 '오늘', 이후는 요일+날짜 표시 |
| 3 | 시간 슬롯 그리드 | 선택한 날짜의 예약 가능 시간대를 30분 단위로 전체 노출 (24시간제). 예약 불가 슬롯은 비활성 처리 |
| 4 | 확인 바텀시트 | 선택한 시간·날짜 확정. `trial_free` ON: 바로 예약, OFF: 결제 포함 |

**예약 슬롯 표시 규칙:**
- 레슨은 30분 단위로 열리며, 시간은 24시간제로 표시 (예: 20:00, 20:30, 21:00)
- 예약 가능 범위: 오늘 포함 7일
- 예약 불가(튜터 없음·이미 예약됨) 슬롯은 비활성 상태로 표시
- 선택 시 날짜+시간 조합이 확인 바텀시트에 표시

> 📊 **데이터 근거** (ClickHouse, MySQL 교차 검증): 예약 피크 시간대는 **22시(1,101건) > 23시(1,092건) > 21시(961건) > 19시(858건)** 순. 예약-수업 시간 갭은 1-6시간이 최다(31.3%)이며, **86.3%가 24시간 내** 수업 참석. 요일별로는 화·수 예약이 가장 많음. 상세 분석: [[Clickhouse 분석 데이터]]

**예약 실패 처리:**
- 시간 선택 후 확인 단계에서 해당 슬롯이 소진된 경우, "선택한 시간이 마감됐어요. 다른 시간을 선택해주세요" 메시지 노출 후 시간 그리드 갱신

### 건너뛰기 정책

> **Intro + Step 1~4: 건너뛰기 없음.** Intro 포함 전체 소요 시간 ~2분 15초의 라이트한 플로우이므로 건너뛰기 옵션을 제공하지 않는다.
> **로그인: 건너뛰기 없음.** Step 5(예약)는 서버 API 호출이 필요하므로 로그인 필수. Step 4 "무료로 시작하기" CTA가 유일한 진행 경로.
> **Step 5만 "나중에 할게요" 제공.** 체험 예약은 유저 의사 존중. 건너뛴 유저에게는 홈 배너로 재유도.
> **알림 권한: 거부 가능.** 거부해도 예약 완료 화면으로 정상 진행.

> **v1.0 변경**: 온보딩 진행 상태(`GET/PUT /status`)는 localStorage로 관리. ~1분 내외 플로우이므로 서버 저장 불필요. 서버 부하 최소화 우선. v2에서 크로스 디바이스 복구가 필요해지면 서버 API 추가.

**데이터 모델:**

```
onboarding_result {
  user_id          -- FK
  completed_at     -- 온보딩 완료 시각 (nullable)
  language         -- en / ja
  goals            -- JSON array ["business", "travel", ...]
  recommended_level -- 추천 시작 레벨 (nullable, 잠금 아닌 추천)
  skipped_trial    -- boolean (Step 5에서 "나중에" 선택 여부)
  created_at
  updated_at

## 7. A/B 테스트 계획

### 실험 설계

| 항목        | 내용                                                                                                  |
| --------- | --------------------------------------------------------------------------------------------------- |
| 실험 단위     | 유저 (신규 가입자 — 체험 전 신규만 대상)                                                                           |
| 할당        | 피처 플래그 기반 랜덤 배정                                                                                     |
| 최소 표본 크기  | 각 그룹 200명 이상 (유의수준 5%, 검정력 80% 기준)                                                                  |
| 일평균 신규 가입 | **~118명/일** (MySQL 실측, 2026-01-01~03-04 기준 7,452명/63일). 각 그룹 200명 도달에 **약 3.4일**, 1,000명은 **약 17일** |
| 실험 기간     | 최소 2주 (표본 확보 + 행동 패턴 안정화 기간)                                                                        |

| 그룹 | 플래그 상태 | 경험 |
|---|---|---|
| Control | `onboarding_v1=OFF` | 기존 플로우 (로그인 → `/subscribes/trial`) |
| Treatment | `onboarding_v1=ON` | 풀 온보딩 (Step 1~5, 시작 레벨 추천 포함) |

> **왜 2그룹인가?** 유저가 이미 모든 레벨/주제를 자유 선택할 수 있고, 첫 수업 후 AI 리포트로 정밀 레벨링이 되므로, 레벨 추천 생략(자가선택 유지) 그룹을 별도로 둘 실익이 없다.

### 의사결정 기준

- Treatment가 Control 대비 **무료체험 신청률 +10%p 이상** → 전체 롤아웃 (Sub 1의 직접 영향 범위가 가입→체험 신청 구간이므로, 1차 성공 지표는 무료체험 신청률)
- 신청률은 올랐지만 체험 완료율이 동일 → Sub 2 개선에 집중
- 유의미하지 않음 → 온보딩 소요시간/단계별 이탈 분석 후 개선

## 12. Sub-PRD 2: 예약 → 체험 완료 (대기 경험 + 노쇼 방지)

**무료 체험 1회 유지** — 기존과 동일. v0.7에서 검토했던 3회 정책은 노쇼의 주원인이 CTA 버그로 확인됨에 따라 기각. 버그 수정 후에도 노쇼율이 높을 경우, 체험 횟수 확대를 재검토한다.

## 13. Sub-PRD 3: 체험 완료 → 결제 유도

### v1 MVP: 인앱 리포트 내 결제 CTA

|---|---|
| 리포트 하단 CTA | "지금 시작하면 첫 수강 혜택을 받을 수 있어요" → 수강권 구매 페이지 이동 |
| 홈 화면 | 기존 홈 화면 유지 (별도 재구성 없음) |

### v2 검토 항목
