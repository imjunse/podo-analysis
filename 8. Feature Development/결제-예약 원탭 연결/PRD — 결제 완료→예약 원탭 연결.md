# PRD: 결제 완료→예약 원탭 연결

## 1. 요약

PODO 서비스에서 결제를 완료한 유저가 첫 수업을 예약하지 않고 이탈하는 문제를 해결한다. 결제 성공 화면의 CTA를 "첫 수업 예약하기"로 변경하여, 결제 직후 가장 동기가 높은 순간에 예약으로 바로 연결한다. 개발 난이도가 낮은 퀵윈이면서, 결제 퍼널 69% 이탈과 1회차 이탈 82.6%를 동시에 개선할 수 있는 ROI 최고 과제다.

---

## 2. 배경

### 맥락: 무엇에 관한 건가?

PODO는 온라인 영어 회화 서비스로, 유저가 수강권을 결제한 뒤 직접 수업을 예약해야 수강이 시작된다. 그런데 결제 완료 후 예약까지 연결하는 경로가 **아예 없다.**

### 왜 지금인가?

정기결제 이탈 분석과 ClickHouse 유저 행동 패턴 분석에서 **결정적 데이터**가 나왔다:

| 지표 | 수치 | 의미 |
|------|:----:|------|
| 결제 퍼널 이탈률 | **69%** | 결제 과정에서 10명 중 7명 이탈 |
| 1회차 이탈률 | **82.6%** | 첫 수업도 안 하고 떠나는 비율 |
| 1~2회차 이탈 비중 | **57%** | 전체 이탈의 과반이 초기에 발생 |
| M3 잔존율 | **45%** | 3개월 시점에 절반 이하만 남음 |

핵심 원인: 돈을 낸 직후가 유저 동기가 가장 높은 순간인데, "다음에 뭘 해야 하지?"를 유저에게 떠넘기고 있다.

### 왜 가능해졌는가?

기술적으로 새로운 건 없다. 기존 예약 화면(`/booking`)과 레슨 선택 화면(`/lessons/regular`)이 이미 있고, CTA 하나를 바꾸는 수준이다. **데이터가 문제를 명확히 보여준 것**이 트리거다.

---

## 3. 목표

### 왜 중요한가?

결제 후 첫 수업까지의 연결이 끊겨 있는 것은 PODO 성장의 가장 큰 병목이다. 매출이 발생해도 유저가 서비스를 경험하지 못하면 해지로 이어진다. 이 한 가지를 해결하면 **신규 유저 첫 7일 여정**(결제→예약→온보딩→넛지)의 첫 단추를 끼운다.

### 전략 정렬

> 신규 유저 첫 7일 여정 설계([[신규 유저 첫 7일 여정 설계]])의 **1단계** 퀵윈.
> 1+2+4번 기능 묶음(원탭 예약 → 온보딩 → 넛지)의 시작점이다.

### Key Results (SMART OKR)

| KR | 현재 | 목표 | 기간 | 측정 방법 |
|----|:----:|:----:|:----:|----------|
| **KR1**: 결제 성공→레슨 선택 진입률 | 미측정 | **70%+** | 배포 후 2주 | `referrer=payment-success` GA 이벤트 |
| **KR2**: 결제 성공→예약 완료 전환율 | 미측정 | **50%+** | 배포 후 2주 | 결제 성공 세션 내 예약 완료 이벤트 |
| **KR3**: 결제 당일 첫 수업 예약률 | 미측정 | **60%+** | 배포 후 1개월 | 결제일-첫 예약일 차이 분석 |
| **KR4**: 1회차 이탈률 | 82.6% | **70% 이하** | 배포 후 2개월 | 기존 이탈 분석 파이프라인 |

### 관찰 지표

- Secondary CTA("나중에 할게요") 클릭률 — 30% 이상이면 CTA 문구/위치 재검토
- 레슨 선택 화면 이탈률 — 높으면 B안(추천 레슨 자동 선택) 검토 트리거

## 4. 대상 유저

### 핵심 세그먼트

**"방금 결제한 신규 유저"** — 수강권을 처음 구매하고, 아직 첫 수업을 예약하지 않은 사람.

| 속성 | 설명 |
|------|------|
| **문제** | "돈을 냈는데, 다음에 뭘 해야 하는지 모르겠다" |
| **동기** | 결제 직후라 서비스 사용 의지가 가장 높은 순간 |
| **현재 경험** | 결제 완료 화면에서 "예습하러 가기"(= 레슨 목록) 또는 "홈으로 이동" 중 선택 |
| **접점** | 모바일 앱, 모바일 웹, 데스크톱 — 3개 플랫폼 |

### 제외 대상

| 유저 | 이유 |
|------|------|
| **체험권 결제 유저** | 체험 수업은 결제 시 자동 배정됨. 별도 분기 처리 |
| **재구매 유저** (이미 수업 이력 있음) | 예약 경험이 있으므로 "첫 수업 예약" 문구 부적합. "수업 목록 보기"로 분기 |
| **기존 유저** (온보딩 소급 미적용) | 이미 서비스 사용 중이므로 이번 스코프 외 |

## 5. 가치 제안

### 유저 관점 — 어떤 "할 일(Job)"을 해결하는가?

> **"수강권을 샀으니, 바로 첫 수업을 잡고 싶다."**

| 관점 | AS-IS (현재) | TO-BE (변경 후) |
|------|-------------|----------------|
| **고객이 얻는 것** | 결제 확인만 됨 | 결제 직후 바로 첫 수업 예약까지 완료 |
| **고객이 피하는 것** | 홈에서 예약 메뉴를 직접 찾아야 함 | 탐색 없이 한 번 탭으로 예약 플로우 진입 |
| **감정적 가치** | "돈 냈는데 뭐 해야 하지?" 불안 | "다음 할 일이 명확하다" 안심 |

### 비즈니스 관점

| 가치 | 설명 |
|------|------|
| **즉각적 매출 보호** | 결제 후 이탈을 줄여 수강권이 실제 사용으로 이어지게 함 |
| **리텐션 기반** | 첫 수업을 빨리 경험할수록 M3 잔존율 상승 |
| **후속 기능 토대** | 온보딩 퍼널(Squad C)의 진입점 역할 |

### 경쟁 우위

현재 PODO의 결제 후 경험은 경쟁 서비스 대비 열위. 대부분의 구독형 에듀테크 서비스는 결제 직후 첫 세션 예약 또는 자동 배정을 제공한다. 이 기능은 **업계 기본 수준으로의 복원**이며, 이후 온보딩+넛지를 붙이면 차별화 포인트가 된다.

## 6. 솔루션

### 6.1 현재 상태 (AS-IS) — 3개 플랫폼 결제 성공 화면

#### 모바일 앱

```
┌─────────────────────────┐
│        결제완료           │
│      🎉 PODO 캐릭터       │
│   "구매가 완료되었어요!"    │
│  "레슨 예약을 위해         │
│   예습하러 가볼까요?"      │
│  ┌─────────────────────┐ │
│  │    예습하러 가기      │ │  ← Primary CTA → /subscribes (레슨 목록)
│  └─────────────────────┘ │
│  ┌─────────────────────┐ │
│  │   홈으로 이동하기     │ │  ← Secondary CTA → /home
│  └─────────────────────┘ │
└─────────────────────────┘
```

**문제**: "예습하러 가기"가 레슨 목록으로 이동. 예약 화면이 아님.

#### 모바일 웹

```
┌─────────────────────────┐
│        결제완료           │
│      🎉 PODO 캐릭터       │
│   "구매가 완료되었어요!"    │
│  "레슨 준비를 위해         │
│   앱으로 이동해 볼까요?"   │
│  ┌─────────────────────┐ │
│  │   앱으로 이동하기     │ │  ← Primary CTA → 딥링크 (podo://new_home_v1.php)
│  └─────────────────────┘ │
│  ┌─────────────────────┐ │
│  │   홈으로 이동하기     │ │  ← Secondary CTA → /home
│  └─────────────────────┘ │
└─────────────────────────┘
```

**문제**: 앱 홈으로 이동할 뿐, 예약 경로 안내 없음.

#### 데스크톱

```
┌──────────────────────────────────┐
│    "구매가 완료되었어요!           │
│     레슨 준비를 위해              │
│     앱으로 이동해 볼까요?"        │
│          ┌─────────┐              │
│          │ QR CODE │              │
│          └─────────┘              │
│      ┌──────────────────┐         │
│      │  홈으로 이동하기   │         │  ← 유일한 CTA → /home
│      └──────────────────┘         │
└──────────────────────────────────┘
```

**문제**: 웹에서 바로 예약할 수 있는 경로 없음.

### 6.2 목표 상태 (TO-BE) — 유저 플로우

```
결제 완료
    ↓
결제 성공 화면
    "구매가 완료되었어요!"
    "첫 수업을 예약해 보세요" ← 안내 문구 변경 (APP/WEB 통합)
    ↓
[Primary CTA]
    APP: "첫 수업 예약하기" ← 신규
    WEB: "앱으로 이동하기" (기존 유지)
[Secondary CTA] "나중에 할게요" ← 변경 (APP/WEB 공통)
    ↓
레슨 선택 화면 (/lessons/regular?referrer=payment-success)
    "첫 수업을 골라보세요" 안내 배너 ← 조건부 노출
    ↓
레슨 카드 선택 → 예약 화면 진입
    (/booking?classId={id}&type=new&classType=regular&referrer=payment-success)
    ↓
예약 완료
    ↓
(→ 후속: 온보딩 퍼널 연결 — Squad C 스코프)
```

#### 플랫폼별 CTA 변경

| 플랫폼 | AS-IS Primary CTA | **TO-BE Primary CTA** | AS-IS Secondary CTA | **TO-BE Secondary CTA** |
|--------|--------------------|-----------------------|---------------------|------------------------|
| **모바일 앱** | "예습하러 가기" → `/subscribes` | **"첫 수업 예약하기"** → `/lessons/regular?referrer=payment-success` | "홈으로 이동하기" → `/home` | **"나중에 할게요"** → `/subscribes` |
| **모바일 웹** | "앱으로 이동하기" → 딥링크 | **변경 없음** (앱 유도 유지) | "홈으로 이동하기" → `/home` | **"나중에 할게요"** → `/subscribes` |
| **데스크톱** | QR코드 + "홈으로 이동하기" | **변경 없음** (QR코드 유지) | — | **변경 없음** |

### 6.3 핵심 기능 상세

#### 기능 1: 결제 성공 화면 CTA 변경

**파일**: `apps/web/src/views/subscribe-payment-success/view.tsx`

**변경 사항**:
- Primary CTA: 텍스트 "첫 수업 예약하기", 목적지 `/lessons/regular?referrer=payment-success&curriculumType={결제한수강권타입}`
- Secondary CTA: 텍스트 "나중에 할게요", 목적지 `/subscribes`
- 안내 문구 변경: "첫 수업을 예약해 보세요" (APP/WEB 통합, 기존 플랫폼별 문구 대체)
- 기존 플랫폼 분기 로직(`from`, `maxTouchPoints`) 유지

**변경하지 않는 것**: 축하 애니메이션, GA4 purchase 이벤트, 결제 정보 표시

#### 기능 2: referrer 파라미터 전파

**파일**: `packages/navigation/src/builders/booking-url.ts`

`referrer=payment-success` 파라미터를 예약 완료까지 전달하여:
- 레슨 선택 화면에서 "첫 수업을 골라보세요" 안내 배너 조건부 노출
- 예약 완료 후 뒤로가기 경로를 홈으로 변경
- 퍼널 분석용 GA 이벤트 트래킹

#### 기능 3: GA4 이벤트 추가

`payment_success_cta_click` 이벤트:
- `cta_type`: "book_first_class" | "later" | "home"
- `platform`: "app" | "mobile_web" | "desktop"
- `curriculum_type`: 결제한 수강권 타입

### 6.4 엣지 케이스

| 케이스 | 처리 방안 |
|--------|----------|
| **체험권 결제 유저** | 체험 수업은 결제 시 자동 배정(`extras.classId` 반환). "예약하기" 대신 "수업 확인하기" → `/reservation` |
| **재구매 유저** (기존 예약 있음) | "첫 수업 예약하기" 대신 "수업 목록 보기" → `/subscribes` |
| **딥링크 미지원** | 폴백: 기존 앱 홈 딥링크 유지 + 앱 진입 시 인앱 배너로 예약 유도 |
| **예약 가능 시간 없음** | 레슨 선택 화면에서 기존 "예약 불가" 처리 그대로 동작 |

```
│   └── (경로 확인 필요)
└── 예약 완료 화면 ← 뒤로가기 경로 변경
    └── (경로 확인 필요)
```

## 핸드오프 포인트

> [[신규 유저 첫 7일 여정 설계]] 참조

이 기능의 **끝점**은 "예약 완료 화면"이다.

예약 완료 후 Squad C의 온보딩 퍼널로 연결되는 시점은 온보딩 PRD에서 정의한다. 이 PRD에서는 `referrer=payment-success` 파라미터를 예약 완료까지 전달하여, 온보딝 퍼널이 이 파라미터를 기반으로 트리거할 수 있도록 준비한다.
