# PRD: 결제 완료→예약 원탭 연결

## 1. 요약

| 항목 | 내용 |
|------|------|
| **한 줄 요약** | 결제 성공 화면에서 첫 수업 예약으로 바로 연결한다 |
| **문제** | 결제 완료 후 예약까지의 연결이 없어 유저가 이탈한다 |
| **해결** | 결제 성공 화면의 Primary CTA를 "첫 수업 예약하기"로 변경 |
| **스쿼드** | Squad A — 전환 퍼널 (PO: 박선영) |
| **난이도** | 낮 (퀵윈) |
| **목표 배포** | 3M 전반 |

## 2. 문제 정의

### 데이터 근거

| 지표 | 수치 | 출처 |
|------|:----:|------|
| 결제 퍼널 이탈률 | **69%** | ClickHouse 유저 행동 패턴 분석 |
| 1회차 이탈률 | **82.6%** | 정기결제 이탈 분석 |
| 1~2회차 이탈이 전체 이탈에서 차지하는 비중 | **57%** | 정기결제 이탈 분석 |

### 현재 상태 (AS-IS)

결제 성공 화면(`/subscribes/payment/{subscribeId}/success`)의 현재 CTA:

> 아래 화면 캡처는 Figma `[PODO] APP DESIGN` 파일 기준이며, 코드 분석 결과와 **동일함을 확인**함.

```

│        결제완료           │
│                          │
│      🎉 PODO 캐릭터       │
│                          │

│                          │
│  ┌─────────────────────┐ │
│  │    예습하러 가기      │ │  ← Primary CTA → /subscribes (레슨 목록)
│  └─────────────────────┘ │
│  ┌─────────────────────┐ │
│  │   홈으로 이동하기     │ │  ← Secondary CTA → /home
```

**문제**: "예습하러 가기"가 레슨 목록(`/subscribes`)으로 이동. 예약 화면이 아님. 유저가 레슨 목록에서 다시 수업을 찾아 예약까지 해야 하는 추가 탐색 필요.

**핵심 문제**: 돈을 낸 직후가 유저의 동기(motivation)가 가장 높은 순간인데, 3개 플랫폼 모두 "다음에 뭘 해야 하지?"를 유저에게 떠넘기고 있다. 예약으로 가는 직접 경로가 어디에도 없다.

### 현재 코드 구조

```
파일 위치:
├── 결제 성공 페이지 (라우트)
│   └── apps/web/src/app/(internal)/subscribes/payment/[subscribeId]/success/page.tsx
├── 결제 성공 뷰 (컴포넌트)
│   └── apps/web/src/views/subscribe-payment-success/view.tsx
├── 예약 페이지 (라우트)
│   └── apps/web/src/app/(internal)/booking/page.tsx
├── 예약 뷰 (컴포넌트)
│   └── apps/web/src/views/booking/view.tsx
└── 예약 URL 빌더
    └── packages/navigation/src/builders/booking-url.ts
```

**예약 화면 진입 조건**: `/booking?classId={classId}&type=new&classType=regular`
- `classId`: 필수 — 레슨 카드에서 선택한 수업 ID
- `type`: `new` (신규 예약) 또는 `edit` (변경)
- `classType`: `regular` 또는 `trial`

## 3. 목표 상태 (TO-BE)

### 유저 플로우

```
결제 완료
    ↓
결제 성공 화면
    "구매가 완료되었어요!"
    ↓
[Primary CTA] "첫 수업 예약하기" ← 신규
[Secondary CTA] "홈으로 이동하기" ← 기존 유지
    ↓
예약 화면 진입
    ↓
수업 예약 완료
    ↓
(→ 후속: 온보딩 퍼널 연결 — Squad C 스코프)
```

### 플랫폼별 CTA 변경

| 플랫폼 | AS-IS Primary CTA | **TO-BE Primary CTA** | TO-BE Secondary CTA |
|--------|--------------------|-----------------------|---------------------|
| **모바일 앱** | "예습하러 가기" → `/subscribes` | **"첫 수업 예약하기"** → 예약 플로우 진입 | "나중에 할게요" → `/subscribes` |
| **모바일 웹** | "앱으로 이동하기" → 딥링크 | **"첫 수업 예약하기"** → 앱 딥링크(예약 화면) | "앱 다운로드" → 스토어 |
| **데스크톱** | QR코드 + "홈으로" | **"첫 수업 예약하기"** → 예약 화면 | "홈으로 이동하기" → `/home` |

## 4. 상세 설계

### 4-1. 모바일 앱: 예약 플로우 진입

**현재 예약 진입의 문제:**
예약 화면(`/booking`)은 `classId`가 필수 파라미터인데, 결제 성공 시점에는 아직 수업을 선택하지 않은 상태.

**해결 방안:**

| 방안 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **A안: 레슨 선택 화면 경유** | "첫 수업 예약하기" → `/lessons/regular` (레슨 선택) → 레슨 선택 → `/booking` | 기존 플로우 재활용, 개발 최소 | 한 단계 추가됨 |
| **B안: 추천 레슨 자동 선택** | 결제한 수강권 타입 + 유저 레벨 기반으로 첫 수업 자동 추천 → `/booking?classId={추천classId}` | 원탭 경험 | 추천 로직 필요, 스코프 증가 |
| **C안: 레슨 선택 내장** | 결제 성공 화면 하단에 추천 레슨 3개 카드 노출 → 선택 시 `/booking` 직행 | 한 화면에서 해결 | UI 변경 범위 큼 |

> **추천: A안** — 퀵윈 취지에 맞게 개발 범위 최소화. `/lessons/regular`로 보내되, 상단에 "첫 수업을 골라보세요" 안내 배너 추가.

**A안 구현 시 라우팅:**
```
결제 성공 화면
    ↓ "첫 수업 예약하기" 클릭
    ↓
/lessons/regular?referrer=payment-success&curriculumType={결제한수강권타입}
    ↓ 레슨 카드 "예약하기" 클릭
    ↓
/booking?classId={classId}&type=new&classType=regular&referrer=payment-success
    ↓ 예약 완료
    ↓
예약 완료 화면 (→ 후속 온보딩 연결 포인트)
```

**`referrer=payment-success` 파라미터 추가 이유:**
- 레슨 선택 화면에서 "첫 수업을 골라보세요" 안내 배너를 조건부 노출
- 예약 완료 후 뒤로가기 경로를 `/subscribes` 대신 홈으로 변경
- 퍼널 분석을 위한 GA 이벤트 트래킹

### 4-2. 모바일 웹: 앱 딥링크

현재 딥링크: `podo://new_home_v1.php` (홈)

**변경**: `podo://lessons/regular?referrer=payment-success` (레슨 선택 화면)

> 딥링크 스킴이 레슨 선택 화면을 지원하는지 확인 필요. 미지원 시 홈 딥링크 유지하되, 앱 진입 시 인앱 메시지로 "첫 수업 예약하기" 유도.

### 4-3. 데스크톱: 웹 예약

데스크톱에서도 `/lessons/regular?referrer=payment-success`로 이동.
QR코드는 Secondary CTA 영역으로 이동.

### 4-4. 결제 성공 화면 UI 변경 사항

**변경 범위** (파일: `views/subscribe-payment-success/view.tsx`):

1. Primary CTA 텍스트 및 라우팅 변경
2. Secondary CTA 텍스트 변경 ("나중에 할게요")
3. 성공 메시지 하단에 한 줄 안내 추가: "수업을 예약해야 수강이 시작돼요"
4. 플랫폼별 분기 로직은 기존 구조(`from`, `maxTouchPoints`) 유지

**변경하지 않는 것:**
- 축하 애니메이션 (PartyPodoIconLottie)
- GA4 purchase 이벤트 발행 로직
- 결제 정보 표시 영역

## 5. 엣지 케이스

| 케이스 | 처리 방안 |
|--------|----------|
| **체험권 결제 유저** | 체험 수업은 결제 시 자동 배정됨 (`extras.classId` 반환). "예약하기" 대신 "수업 확인하기" → `/reservation`으로 이동 |
| **이미 예약이 있는 유저** (수강권 추가 구매) | "첫 수업 예약하기" 대신 "수업 목록 보기" → `/subscribes`로 이동 |
| **딥링크 미지원 시** | 폴백: 기존 앱 홈 딥링크 유지 + 앱 진입 시 인앱 배너로 예약 유도 |
| **예약 가능 시간이 없는 경우** | 레슨 선택 화면에서 기존 "예약 불가" 처리 그대로 동작 |

## 6. 성공 지표

| 지표 | 현재 | 목표 | 측정 방법 |
|------|:----:|:----:|----------|
| 결제 성공→레슨 선택 진입률 | 미측정 | **70%+** | `referrer=payment-success` 파라미터 기반 GA 이벤트 |
| 결제 성공→예약 완료 전환율 | 미측정 | **50%+** | 결제 성공 세션 내 예약 완료 이벤트 |
| 결제 당일 첫 수업 예약률 | 미측정 | **60%+** | 결제일과 첫 예약일 차이 분석 |
| 1회차 이탈률 | 82.6% | **70% 이하** | 기존 이탈 분석 파이프라인 |

### 관찰 지표 (직접 목표는 아니지만 모니터링)

- Secondary CTA("나중에 할게요") 클릭률 — 30% 이상이면 CTA 문구/위치 재검토
- 레슨 선택 화면 이탈률 — 높으면 B안(추천 레슨) 검토 트리거

## 9. 핸드오프 포인트

> [[신규 유저 첫 7일 여정 설계]] 참조

이 기능의 **끝점**은 "예약 완료 화면"이다.

예약 완료 후 Squad C의 온보딩 퍼널로 연결되는 시점은, 온보딩 PRD에서 정의한다. 이 PRD에서는 **`referrer=payment-success` 파라미터를 예약 완료까지 전달**하여, 온보딩 퍼널이 이 파라미터를 기반으로 트리거할 수 있도록 준비한다.
