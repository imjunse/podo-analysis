# PODO UIUX 개선 — Ontology 검증 완료

> 기존 [[PODO 사용자 플로우 핵심 개선 포인트]]를 **podo-docs 3-관점 온톨로지**(도메인/프론트엔드/백엔드)로 교차 검증한 결과입니다.
> 검증일: 2026-03-05

---

## P1. 결제 → 예약 자동 연결 부재

| | 내용 |
|---|------|
| **As-Is** | 결제 완료 화면(`/subscribes/payment/{id}/success`)에 "홈으로 가기" 버튼만 존재. 사용자가 직접 홈 → 수업 탭 → 예약 페이지로 이동해야 함 |
| **To-Be** | 결제 완료 시 Ticket 생성 직후 "바로 예약하기" CTA 제공 → `/booking?classId={id}&type=new`로 직행 |
| **Ontology 근거** | `subscription-flow.md`: 결제 5단계에서 Ticket 생성 시 classId 확보 가능. `user-flows.md`: 결제 성공 페이지에 홈 이동만 존재 |
| **난이도** | 낮음 (프론트엔드 CTA 추가 + 라우팅) |
| **영향도** | 높음 |

> [!tip] 퀵윈 가능
> Ticket 생성 API 응답에 이미 `ticketId`와 `langType`이 포함되므로, 결제 성공 화면에서 바로 `/booking` 라우트로 연결 가능.

**연관**: [[정기결제 이탈 심층 분석]] — 1~2회차 이탈률이 82~96%로 심각. 첫 결제→첫 수업 연결이 빠를수록 잔존율 개선 기대.

## P2. 결제 단계 과다

| | 내용 |
|---|------|
| **As-Is** | 구독목록 → 상품선택 → 결제페이지 → (정기구독 시) 결제수단 등록 **별도 페이지** (`/my-podo/payment-methods/register`) → 복귀 → 약관동의 → 결제실행 → 폴링 → 성공 |
| **To-Be** | 결제수단 등록을 결제 플로우 내 **인라인 바텀시트**로 처리. 페이지 이탈 없이 등록 → 결제 한 번에 완료 |
| **Ontology 근거** | `user-flows.md`: 정기구독 결제 시 결제수단 미등록이면 `/my-podo/payment-methods/register`로 페이지 이동 필요 — 이탈 포인트. `feature-flags.md`: `page_subscribe_ticket_v2_enabled` 플래그로 V2 결제 페이지 분기 중 |
| **난이도** | 중간 (결제수단 등록 UI를 모달/바텀시트로 분리) |
| **영향도** | 높음 |

**연관**: [[PODO 매출-리텐션 심층분석 (2026-02)]] — 신규결제 건수 자체가 감소 추세. 결제 퍼널 간소화가 전환율에 직접 영향.

## P3. 홈 화면 분리

| | 내용 |
|---|------|
| **As-Is** | `/home`(Basic)과 `/home/ai`(Smart Talk) 두 개의 홈이 물리적 분리. `HomeRedirection`이 구독 타입에 따라 분기. 사용자가 다른 홈을 보려면 수동 전환 필요 |
| **To-Be** | 단일 통합 홈에서 구독 상태에 따라 섹션을 동적 렌더링. 예약된 수업 바로가기 + Smart Talk 섹션 + AI Learning 섹션 + 일반 수업 섹션 |
| **Ontology 근거** | `feature-flags.md`: `enable_react_home` — Nuxt→React 마이그레이션 진행 중. 전환 시점에 통합하면 이중 작업 방지. `user-flows.md`: 리다이렉트 로직에서 HOME/AI_LEARNING 분기 확인 |
| **난이도** | 높음 (홈 아키텍처 재설계 + 피처플래그 전환) |
| **영향도** | 높음 |

> [!warning] 타이밍
> React 마이그레이션과 동시에 진행해야 효율적. 별도로 하면 이중 작업 발생.

## P4. 온보딩 플로우 부재

| | 내용 |
|---|------|
| **As-Is** | OAuth 로그인 성공 → destination에 따라 바로 리다이렉트. 레벨테스트, 학습목표, 서비스 안내 없음. 피처플래그에도 온보딩 관련 항목 없음 |
| **To-Be** | 최초 로그인 시 온보딩 퍼널: 레벨테스트 → 레벨 선택 → 첫 수업 부스터 안내 → 홈 진입. 2회차부터 바이패스 |
| **Ontology 근거** | `user-flows.md`: 로그인 후 리다이렉트 로직에 `첫 방문` 체크 없음. `subscription-flow.md`: TRIAL 체험권 존재하나 학습 레벨 설정 플로우 부재. `feature-flags.md`: 온보딩 관련 플래그 0개 |
| **난이도** | 중간 (온보딩 퍼널 신규 개발 + 첫 방문 판별 로직) |
| **영향도** | 높음 |

**연관**: [[정기결제 이탈 심층 분석]] — 1회차 이탈률 82.6%. 온보딩으로 서비스 가치 체험 없이 바로 결제→이탈 패턴 반복.

## P5. 리텐션 오퍼 정적

| | 내용 |
|---|------|
| **As-Is** | WelcomeBack 프로모션이 단일 유형(`WELCOMEBACK_30`). 이탈 사유와 무관하게 동일 할인 오퍼. 풀페이지 팝업 + 배너로 노출 |
| **To-Be** | `le_subscribe_mapp_history`의 해지 사유 데이터를 활용한 맞춤형 오퍼. 가격 불만 → 할인, 시간 부족 → 일시정지 안내, 품질 불만 → 1:1 상담 연결 등 |
| **Ontology 근거** | `subscription/policies.md`: 해지 시 이력 기록 필수 (`le_subscribe_mapp_history`). 사유 데이터 이미 존재하나 리텐션 오퍼에 미활용. `user-flows.md` 6장: WelcomeBack 프로모션은 `promotionType` 단일 분기 |
| **난이도** | 중간 (오퍼 로직 분기 + 프로모션 타입 확장) |
| **영향도** | 중간 |

**연관**: [[PODO 매출-리텐션 심층분석 (2026-02)]] — 정기결제액 51% 감소(Leaky Bucket). 이탈 사유 기반 맞춤 오퍼가 정기결제 잔존율에 직접 영향.

## P6. 동일 언어 중복 구매 제한

| | 내용 |
|---|------|
| **As-Is** | 동일 언어 타입의 활성 구독이 있으면 추가 구매 에러. 멀티트랙(2개+ 구독) 보유 시 보너스 지급도 자동 제외. PRD는 이미 작성 완료 |
| **To-Be** | 동일 언어 중복 구매 허용. 고객 주도 활성화(activation) 방식으로 전환. 멀티트랙 보너스 제외 규칙도 함께 수정 |
| **Ontology 근거** | `GT_SUBSCRIBE_MAPP.md`: USER_ID+SUBSCRIBE_ID 매핑에서 STATUS 활성 체크로 중복 방지. `ticket-policy.md`: 멀티트랙 제외 규칙(2개+ 활성 구독 시 보너스 지급 제외, DoubleTrack ENJP만 예외) → 동시 수정 필요. `subscription-flow.md`: "동일 언어 타입의 활성 구독이 이미 있으면 에러. 예외: 연장 구독은 허용" |
| **난이도** | 높음 (백엔드 정책 변경 + 보너스 로직 수정 + 관리자 화면) |
| **영향도** | 중간 |

> [!info] PRD 존재
> `prd/duplicate-ticket-purchase-and-retention-promotion.md`에 상세 PRD 작성 완료. 구현 단계로 넘어갈 수 있음.

## P7. 에러 메시지 영어

| | 내용 |
|---|------|
| **As-Is** | ErrorBoundary에서 "Something went wrong" / "An error occurred" 영어 에러 표시. 한국인 사용자 대상 서비스에서 부자연스러움 |
| **To-Be** | 에러 메시지 한글화. "문제가 발생했어요" / "잠시 후 다시 시도해 주세요" 등으로 교체 |
| **Ontology 근거** | `home.md` 분석에서 하드코딩된 영어 문구 확인. i18n 시스템 미도입. Ontology 문서에서 에러 메시지 국제화 관련 문서 없음 |
| **난이도** | 낮음 (문자열 교체) |
| **영향도** | 낮음 |

> [!tip] 퀵윈 가능
> P1과 함께 빠르게 처리 가능한 항목.
