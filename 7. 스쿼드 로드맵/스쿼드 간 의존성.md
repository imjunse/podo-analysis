# 스쿼드 간 의존성 맵

## 의존성 관계

```
Squad A (전환)           Squad B (학습경험)        Squad C (온보딩/리텐션)
─────────────           ──────────────          ─────────────────
결제→예약 연결 ─────────→ 예약 플로우 연동
결제→예약(정규) ←──경로분리──→ 온보딩 Step5(체험 예약)
                                                온보딩 Step3 ──→ 기존 퀴즈 인프라 재활용 (신규 API 불필요)
결제 단계 축소                                   맞춤 리텐션 오퍼 ──→ 해지사유 DB 활용
                       학습 플로우 구축           ──────────→ 홈 통합 (6M+)
중복구매 ───────────────────────────────→ 멀티트랙 보너스 규칙
                       AI/Mini Lessons ──→ AI 미니레슨 전환 경로
                       AI 리포트 ←── 온보딩 Step4에서 "첫 수업 후 AI 분석" 예고
```

## 동기화 포인트

### 3M — 예약 진입점 통일

- **Squad A**: 결제 완료→예약 연결 기능 개발
- **Squad B**: 예약 변경 UX 개선 진행
- **합의 필요**: 예약 화면 진입 경로를 통일하여 결제 후 진입과 기존 유저 진입이 동일 UX

### 4M — 온보딩 × AI 리포트 예고

- **Squad C**: 온보딩 Step 4에서 "첫 수업 후 AI가 실력을 분석해줘요" 예고 메시지 노출 (실제 AI 기능 경험은 온보딩에 포함하지 않음)
- **Squad B**: AI 리포트 도달 경로 개선 — 첫 수업 완료 후 AI 리포트로 자연스럽게 연결
- **합의 완료**: 온보딩에서는 예고만, 실제 AI 경험은 첫 수업 후로 확정 (PRD v0.4)

### 4M — 체험 예약(C) ↔ 정규 결제 예약(A) 경로 분리

- **Squad A**: 결제 완료→예약 원탭 연결 (정규 결제 유저 대상)
- **Squad C**: 온보딩 Step 5에서 무료 체험 레슨 예약 (체험 전 신규 유저 대상)
- **합의 필요**: 두 경로가 예약 UI를 공유할 수 있으나, 진입 조건이 다름. 예약 화면 컴포넌트 재활용 범위 확인
- **참고**: 온보딩 v1 스코프는 무료 체험 예약까지. 정규 결제 유도/수강권 추천은 Squad A 담당 (PRD Non-Goal)

### 5M — 포인트 시스템 공동 설계

- **Squad B**: 예습 완료 포인트, AI 리포트 확인 포인트
- **Squad A**: 포인트 설계 (리텐션 인센티브)
- **합의 필요**: 포인트 경제 설계 (획득/사용 규칙, 인플레이션 방지)

### 5M — 리텐션 × 정기전환 연동

- **Squad C**: 맞춤 리텐션 오퍼 발송
- **Squad A**: 정기전환율 개선
- **합의 필요**: 이탈 방지 오퍼가 정기전환 프로모션과 충돌하지 않도록 조율

### 6M — 홈 통합 × 학습 대시보드

- **Squad C**: 홈 화면 통합 (두 개 홈 → 하나)
- **Squad B**: 학습 진도 시각화
- **합의 필요**: 통합 홈에 학습 대시보드를 어떤 형태로 포함할지
